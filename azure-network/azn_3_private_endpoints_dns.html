How Databricks and PaaS services get private IPs inside your VNet — and how DNS resolves them from workers, workspaces, and on-prem

Concept — What a Private Endpoint Actually Is

Azure PaaS service public DNS normally

  • Databricks workspace · adb-123.12.azuredatabricks.net
  • ADLS Gen2 · myacct.dfs.core.windows.net
  • Key Vault · myvault.vault.azure.net
  • Azure SQL · mysrv.database.windows.net
  • These hostnames resolve to public IPs on the internet by default
PRIVATE LINK 🔗 Private Endpoint

Private Endpoint NIC in your VNet

  • A network interface lives in your subnet with a private IP (e.g. 10.139.128.7)
  • Azure injects a CNAME / A-record pointing the public FQDN at the private IP — via Private DNS zone
  • All traffic from the VNet (and peered VNets) to that service now goes over the Azure backbone, never the internet
  • Once public access is disabled on the service, there is no other path
The Three Databricks Private Endpoints You'll Create
Front-End · UI/API

Workspace databricks_ui_api

Sub-resource: databricks_ui_api

The PE users & tools hit to reach the workspace web UI and REST API. Replaces the public adb-NNN.NN.azuredatabricks.net with a private IP in your workspace-pe-subnet.

Once created and public access is disabled, every UI click and every API call from notebooks, Terraform, CI — even databricks-cli — must traverse this PE.

DNS zone: privatelink.azuredatabricks.net

Browser Auth

Workspace browser_authentication

Sub-resource: browser_authentication

Private path for the interactive SSO/OAuth login flow (login.azuredatabricks.net). Needed so the login popup resolves to your private IP rather than the public endpoint.

Without this, even with UI/API PE in place, browser login can fail with DNS or CORS errors when public access is off.

Same DNS zone: privatelink.azuredatabricks.net

Back-End · SCC

Back-end databricks_ui_api

Sub-resource: databricks_ui_api (same type, diff purpose)

The PE that cluster workers use to reach the control plane over the Secure Cluster Connectivity relay. Required when you disable public IPs on workers (NPIP).

Lives in the pe-subnet inside the Databricks spoke VNet so workers can find it without leaving the VNet.

DNS resolves to the private IP via the same privatelink.azuredatabricks.net zone.

DNS Resolution Flow — From On-Prem to Private Endpoint

How your laptop (or an on-prem service) resolves adb-123.12.azuredatabricks.net to a private Azure IP

1
On-prem query

Laptop asks its local DNS (e.g. UDM Pro's resolver) for adb-123.12.azuredatabricks.net.

2
Conditional forwarder

UDM / on-prem DNS has a forwarder rule: "for *.azuredatabricks.net, ask Azure DNS Resolver inbound endpoint at 172.16.4.4."

3
Azure Private DNS Resolver

Inbound endpoint receives the query over the S2S VPN tunnel, forwards to Azure-provided DNS (168.63.129.16).

4
Private DNS zone match

Azure-provided DNS sees the name is a CNAME to *.privatelink.azuredatabricks.net, matches the private zone, returns the Private Endpoint's IP (e.g. 10.139.129.5).

5
Traffic over VPN

Laptop connects to 10.139.129.5. That IP is reachable via the S2S VPN + VNet peering. Request never touches the public internet.

Create private DNS zones (once per service type, in hub RG) az network private-dns zone create -g rg-hub -n privatelink.azuredatabricks.net az network private-dns zone create -g rg-hub -n privatelink.blob.core.windows.net az network private-dns zone create -g rg-hub -n privatelink.dfs.core.windows.net az network private-dns zone create -g rg-hub -n privatelink.vaultcore.azure.net # 3. Link zones to BOTH VNets (hub + dbx spoke) for zone in privatelink.azuredatabricks.net privatelink.blob.core.windows.net privatelink.dfs.core.windows.net privatelink.vaultcore.azure.net; do az network private-dns link vnet create -g rg-hub -n $zone-hub --zone-name $zone --virtual-network vnet-hub --registration-enabled false az network private-dns link vnet create -g rg-hub -n $zone-dbx --zone-name $zone --virtual-network vnet-dbx --registration-enabled false done # 4. Private Endpoint for Databricks UI/API (front-end) WS_ID=$(az databricks workspace show -g rg-dbx -n dbx-prod --query id -o tsv) az network private-endpoint create -g rg-dbx -n pe-dbx-ui \ --vnet-name vnet-dbx --subnet workspace-pe-subnet \ --private-connection-resource-id $WS_ID \ --group-id databricks_ui_api \ --connection-name dbx-ui-conn # 5. Private Endpoint for browser authentication az network private-endpoint create -g rg-dbx -n pe-dbx-auth \ --vnet-name vnet-dbx --subnet workspace-pe-subnet \ --private-connection-resource-id $WS_ID \ --group-id browser_authentication --connection-name dbx-auth-conn # 6. Private Endpoint for back-end (SCC relay) — lives in pe-subnet, not workspace-pe-subnet az network private-endpoint create -g rg-dbx -n pe-dbx-scc \ --vnet-name vnet-dbx --subnet pe-subnet \ --private-connection-resource-id $WS_ID \ --group-id databricks_ui_api --connection-name dbx-scc-conn # 7. Auto-register PE IPs into the zone (one-liner per PE) az network private-endpoint dns-zone-group create -g rg-dbx \ --endpoint-name pe-dbx-ui -n zone-group-ui \ --private-dns-zone privatelink.azuredatabricks.net --zone-name azuredatabricks # 8. Turn off public network access on the workspace — THE LOCK az databricks workspace update -g rg-dbx -n dbx-prod \ --public-network-access Disabled \ --required-nsg-rules NoAzureDatabricksRules

The Missing Link

The privatelink.azuredatabricks.net zone had the right A record (10.139.0.4) but was only linked to databricks-vnet — because the PE auto-created that link when its DNS zone group was attached. Azure Private DNS Resolver (which will live in vnet-hub) cannot see a private zone that isn't linked to its own VNet. Without that link, on-prem queries can never resolve to the private IP — no matter how well the rest of the network is wired.

🛠 The Fix — Zone Link + Delegated Subnet + Resolver + UDM Forwarder

① Link the private DNS zone to vnet-hub (the one that was missing)
az network private-dns link vnet create -g rg-hub -n link-hub \ --zone-name privatelink.azuredatabricks.net \ --virtual-network vnet-hub --registration-enabled false
Verify: az network private-dns link vnet list -g rg-hub -z privatelink.azuredatabricks.net -o table now shows two rows — one per VNet — both Succeeded / Completed.
② Create delegated subnet + DNS Resolver + inbound endpoint
# 2a. /28 subnet DELEGATED to Microsoft.Network/dnsResolvers az network vnet subnet create -g rg-hub --vnet-name vnet-hub -n dns-resolver-in \ --address-prefix 172.16.4.0/28 --delegations Microsoft.Network/dnsResolvers # 2b. Resolver shell attached to vnet-hub az dns-resolver create -g rg-hub -n dns-resolver -l westus3 \ --id $(az network vnet show -g rg-hub -n vnet-hub --query id -o tsv) # 2c. Inbound endpoint — AAZ schema expects FLAT `id:`, not nested `subnet:{id:...}` SUBNET_ID=$(az network vnet subnet show -g rg-hub --vnet-name vnet-hub -n dns-resolver-in --query id -o tsv) az dns-resolver inbound-endpoint create -g rg-hub --dns-resolver-name dns-resolver \ -n in-endpoint -l westus3 \ --ip-configurations "[{private-ip-allocation-method:'Dynamic',id:'$SUBNET_ID'}]" # 2d. Grab the assigned private IP (what UDM will forward to) az dns-resolver inbound-endpoint show -g rg-hub --dns-resolver-name dns-resolver \ -n in-endpoint --query "ipConfigurations[0].privateIpAddress" -o tsv # → 172.16.4.4
⚠ CLI quirk (AAZ-generated az dns-resolver): the --ip-configurations schema is flat — pass the subnet resource ID as id: at the top of each element. Don't wrap it in subnet:{id:...}; older examples use that nested form and now fail with "Model 'AAZObjectArg' has no field named 'subnet'". Check your schema with az dns-resolver inbound-endpoint create --help if in doubt.
③ LAN DNS conditional forwarder (UDM or AdGuard) & verify
# --- Option A — UDM Pro --- # UniFi Network → Settings → Routing → DNS → Conditional forwarder privatelink.azuredatabricks.net → 172.16.4.4 azuredatabricks.net → 172.16.4.4 # --- Option B — AdGuard Home (preferred if you already run it) --- # Settings → DNS settings → Upstream DNS servers (paste at top): [/azuredatabricks.net/privatelink.azuredatabricks.net/]172.16.4.4 [/privatelink.blob.core.windows.net/privatelink.dfs.core.windows.net/]172.16.4.4 [/privatelink.vaultcore.azure.net/privatelink.database.windows.net/]172.16.4.4 # Prereq for both: UDM must also tunnel 172.16.0.0/16 (the hub VNet) over # IPsec so AdGuard / UDM can reach 172.16.4.4. Add it to Remote Networks.
Verify from laptop: nslookup adb-NNN...azuredatabricks.net returns 10.139.0.4 (not the public 4.227.68.192). Flush client DNS cache first: ipconfig /flushdns.

The Second Missing PE — browser_authentication

Databricks exposes two distinct PE sub-resources for browser-based workspace access. databricks_ui_api handles the workspace UI + REST API. browser_authentication handles the SSO / OAuth redirect flow to login.<region>.azuredatabricks.net. API-only clients (PAT + curl) are fine with just UI/API; browsers need both. Without the second PE, the login redirect still lands on the public endpoint, which is now refusing everyone. Same error message, different hop.

🛠 The Second Fix — Create the browser_authentication PE

① Create the PE and wire it into the same DNS zone
WS_ID=$(az databricks workspace show -g <dbx-rg> -n <workspace-name> --query id -o tsv) az network private-endpoint create -g rg-hub -n pe-dbx-browser-auth \ --vnet-name databricks-vnet --subnet default \ --private-connection-resource-id $WS_ID \ --group-id browser_authentication --connection-name dbx-browser-auth-conn # Register the new PE's IP into the same private DNS zone az network private-endpoint dns-zone-group create -g rg-hub \ --endpoint-name pe-dbx-browser-auth -n zone-group-auth \ --private-dns-zone privatelink.azuredatabricks.net --zone-name azuredatabricks
Verify: az databricks workspace show --query "properties.privateEndpointConnections[].groupIds" now returns both ["databricks_ui_api","browser_authentication"], both Approved.
② Flush DNS caches & retest both hops
# AdGuard Home: Settings → DNS → Clear cache # Windows client: ipconfig /flushdns # Both hops must now resolve privately: nslookup adb-NNN...azuredatabricks.net # → 10.139.0.4 (UI/API PE) nslookup login.NN.azuredatabricks.net # → 10.139.0.5 (browser-auth PE) # Open workspace in browser — full OAuth handshake now stays private
Final state: Browser loads workspace URL → Entra ID sign-in page → lands on workspace home. Both hops traverse PEs; nothing touches the public internet. Truly private-only.
⚠ Diagnostic shortcut: if curl -H "Authorization: Bearer <PAT>" https://adb-NNN.../api/2.0/workspace/list works but the browser doesn't — that asymmetry alone is a strong signal that browser_authentication is the missing piece. PAT auth skips the OAuth redirect hop entirely.
