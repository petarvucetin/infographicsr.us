Subnet Delegation — What Makes a Subnet "Databricks Ready"
"delegations": [{ "name": "dbx", "properties": { "serviceName": "Microsoft.Databricks/ workspaces" } }]

Delegating a subnet to Microsoft.Databricks/workspaces

Delegation hands control of a subnet to a specific Azure service so it can inject managed network interfaces (like cluster worker VMs) without you creating them. Both the host and container subnets must be delegated to Microsoft.Databricks/workspaces before the workspace will deploy into them.

A delegated subnet can only hold resources from that service. You cannot mix a Databricks delegation with generic VMs in the same subnet — plan CIDRs accordingly. Other common delegations: Microsoft.Web/serverFarms (App Service), Microsoft.Sql/managedInstances, Microsoft.ContainerInstance/containerGroups.

Set via CLI: az network vnet subnet update ... --delegations Microsoft.Databricks/workspaces

Databricks-Mandated NSG Rules — Attach to Both host-subnet & container-subnet
Azure Databricks publishes a fixed set of NSG rules the workspace requires to function. The rules allow control-plane reachability from workers and intra-VNet cluster comms. Priority must be below 200. Don't modify; add your own allow/deny rules at higher priorities (lower numbers = higher priority in Azure NSGs).

Build It — Azure CLI Step by Step
Step 1 — Create the resource group. All hub and spoke resources share lakeflow-mssql-on-prem, matching the group from the Site-to-Site VPN setup. Set a default location once with az configure --defaults location=<region> so subsequent commands inherit it without repeating the flag.
az group create -n lakeflow-mssql-on-prem
Step 2 — Create the Hub VNet. Address space 10.0.0.0/26 holds all shared-services subnets: VPN Gateway, Azure Firewall, Bastion, and DNS Resolver.
az network vnet create -g lakeflow-mssql-on-prem -n vnet-hub --address-prefix 10.0.0.0/26
Step 3 — Create the Databricks Spoke VNet. 10.0.0.64/26 sits adjacent to the hub within the same 10.0.0.0/25 parent. Workers, container networking, and Private Endpoints all live here.
az network vnet create -g lakeflow-mssql-on-prem -n vnet-dbx --address-prefix 10.0.0.64/26
Step 4 — Create GatewaySubnet. The VPN Gateway must be in a subnet named exactly GatewaySubnet. No NSG may be attached. /28 gives 11 usable IPs, enough for an active/standby gateway pair.
az network vnet subnet create -g lakeflow-mssql-on-prem --vnet-name vnet-hub -n GatewaySubnet --address-prefix 10.0.0.0/28
Step 5 — Create AzureFirewallSubnet. Name must be exactly AzureFirewallSubnet. No NSG allowed. /28 is the practical minimum for a single Firewall instance.
az network vnet subnet create -g lakeflow-mssql-on-prem --vnet-name vnet-hub -n AzureFirewallSubnet --address-prefix 10.0.0.16/28
Step 6 — Create AzureFirewallManagementSubnet. Only needed when Firewall runs in forced-tunnel mode. Azure reserves this subnet for its own management traffic so it bypasses your UDR. Name must be exact. No NSG.
az network vnet subnet create -g lakeflow-mssql-on-prem --vnet-name vnet-hub -n AzureFirewallManagementSubnet --address-prefix 10.0.0.32/28
Step 7 — Create AzureBastionSubnet. Bastion provides browser-based SSH/RDP to private VMs with no public IP. Name must be AzureBastionSubnet exactly. /28 for this lab; production Standard SKU needs at least /26.
az network vnet subnet create -g lakeflow-mssql-on-prem --vnet-name vnet-hub -n AzureBastionSubnet --address-prefix 10.0.0.48/28
Step 8 — Create the DNS Resolver inbound subnet. The inbound endpoint receives privatelink.* queries forwarded from your on-premises UDM Pro, resolving them to private IPs so on-prem clients reach Databricks and storage over the VPN tunnel.
az network vnet subnet create -g lakeflow-mssql-on-prem --vnet-name vnet-hub -n dns-resolver-in --address-prefix 10.0.0.0/28
Step 9 — Create the DNS Resolver outbound subnet. The outbound endpoint forwards queries from Azure resources to your on-premises DNS, so Databricks workers can resolve internal hostnames like your SQL Server.
az network vnet subnet create -g lakeflow-mssql-on-prem --vnet-name vnet-hub -n dns-resolver-out --address-prefix 10.0.0.16/28
Step 10 — Create the host-subnet NSG. Databricks refuses to deploy into a delegated subnet with no NSG. Create it empty — Azure Databricks injects the mandatory control-plane rules automatically when the workspace is provisioned.
az network nsg create -g lakeflow-mssql-on-prem -n nsg-dbx-host
Step 11 — Create the container-subnet NSG. A separate NSG for the container subnet. Both NSGs must exist before the delegated subnet commands run, as those commands reference them by name.
az network nsg create -g lakeflow-mssql-on-prem -n nsg-dbx-container
Step 12 — Create the Databricks host subnet. Carries cluster worker VM primary NICs. Must be delegated to Microsoft.Databricks/workspaces so Azure can inject managed interfaces. Must be the same prefix length as the container subnet.
az network vnet subnet create -g lakeflow-mssql-on-prem --vnet-name vnet-dbx -n host-subnet --address-prefix 10.0.0.64/28 --delegations Microsoft.Databricks/workspaces --network-security-group nsg-dbx-host
Step 13 — Create the Databricks container subnet. Carries container networking interfaces. With Secure Cluster Connectivity, workers have no public IPs — all traffic routes through the SCC relay over Private Endpoint.
az network vnet subnet create -g lakeflow-mssql-on-prem --vnet-name vnet-dbx -n container-subnet --address-prefix 10.0.0.80/28 --delegations Microsoft.Databricks/workspaces --network-security-group nsg-dbx-container
Step 14 — Create the Private Endpoint subnet. PEs for ADLS Gen2, Key Vault, SQL, and the Databricks SCC back-end land here. --private-endpoint-network-policies Disabled is mandatory — Azure blocks PE creation in subnets where this is still active.
az network vnet subnet create -g lakeflow-mssql-on-prem --vnet-name vnet-dbx -n pe-subnet --address-prefix 10.0.0.96/28 --private-endpoint-network-policies Disabled
Step 15 — Create the Workspace PE subnet. Dedicated subnet for Databricks UI/API and browser-auth Private Endpoints. Separating these from back-end PEs keeps NSG rules and diagnostics cleaner.
az network vnet subnet create -g lakeflow-mssql-on-prem --vnet-name vnet-dbx -n workspace-pe-subnet --address-prefix 10.0.0.112/28 --private-endpoint-network-policies Disabled
Step 16 — Capture VNet resource IDs. Peering needs the full Azure resource ID of the remote VNet. CMD uses for /f to capture output into a variable; Bash uses $() subshell. Both set HUB and DBX for the next two steps.
HUB=$(az network vnet show -g lakeflow-mssql-on-prem -n vnet-hub --query id -o tsv)
DBX=$(az network vnet show -g lakeflow-mssql-on-prem -n vnet-dbx --query id -o tsv)
Step 17 — Create Hub to Spoke peering. --allow-gateway-transit lets the spoke borrow the hub VPN Gateway to reach on-premises. --allow-forwarded-traffic passes traffic not originating here, needed when routing through Azure Firewall.
az network vnet peering create -g lakeflow-mssql-on-prem --vnet-name vnet-hub -n hub-to-dbx --remote-vnet $DBX --allow-vnet-access --allow-forwarded-traffic --allow-gateway-transit
Step 18 — Create Spoke to Hub peering. --use-remote-gateways routes the spoke through the hub VPN Gateway. Requires Step 17 to have --allow-gateway-transit already set — otherwise this command fails.
az network vnet peering create -g lakeflow-mssql-on-prem --vnet-name vnet-dbx -n dbx-to-hub --remote-vnet $HUB --allow-vnet-access --allow-forwarded-traffic --use-remote-gateways
Subnet & NSG Gotchas
⚠️

Reserved subnet names are literal

GatewaySubnet, AzureFirewallSubnet, AzureBastionSubnet must be spelled exactly. Azure looks them up by name — a typo = subnet unusable.

📏

Databricks subnets must be equal size

Host and container subnets require the same prefix length. /26 min, /18 max recommended. Azure reserves 5 IPs per subnet (network, broadcast, 3 Azure).

🔗

Delegations are one-way

Once delegated to Databricks, only Databricks-managed resources can live there. You cannot add a generic VM to a Databricks-delegated subnet.

🛑

Don't attach NSGs to reserved subnets

GatewaySubnet, AzureFirewallSubnet, AzureFirewallManagementSubnet must have no NSG. AzureBastionSubnet can have an NSG but with strict Microsoft-specified rules only.

🔒

NSGs are stateful but directional

Allow inbound does not imply allow outbound. Plan both directions. Priority 100 min for Databricks rules; your deny-all should be > 4000 to let everything else evaluate first.

🧮

Service tags beat CIDR lists

Use AzureDatabricks, Storage, Sql, AzureActiveDirectory service tags instead of IP ranges. Microsoft keeps them current; CIDR lists go stale.
