Subnet Delegation — What Makes a Subnet "Databricks Ready"
"delegations": [{ "name": "dbx", "properties": { "serviceName": "Microsoft.Databricks/ workspaces" } }]

Delegating a subnet to Microsoft.Databricks/workspaces

Delegation hands control of a subnet to a specific Azure service so it can inject managed network interfaces (like cluster worker VMs) without you creating them. Both the host and container subnets must be delegated to Microsoft.Databricks/workspaces before the workspace will deploy into them.

A delegated subnet can only hold resources from that service. You cannot mix a Databricks delegation with generic VMs in the same subnet — plan CIDRs accordingly. Other common delegations: Microsoft.Web/serverFarms (App Service), Microsoft.Sql/managedInstances, Microsoft.ContainerInstance/containerGroups.

Set via CLI: az network vnet subnet update ... --delegations Microsoft.Databricks/workspaces

Databricks-Mandated NSG Rules — Attach to Both host-subnet & container-subnet
Azure Databricks publishes a fixed set of NSG rules the workspace requires to function. The rules allow control-plane reachability from workers and intra-VNet cluster comms. Priority must be below 200. Don't modify; add your own allow/deny rules at higher priorities (lower numbers = higher priority in Azure NSGs).

Build It — Azure CLI Step by Step
1

Hub VNet · Databricks Spoke VNet · NSGs · foundation for subnet and peering work

The hub VNet (vnet-hub) and all its subnets were created in the Site-to-Site VPN guide. Verify they exist, then create the Databricks spoke VNet and the two NSGs that Databricks requires on its delegated subnets.

Hub VNet
Created in S2S VPN Guide step 1 — vnet-hub with prefixes 10.0.0.0/26 and 10.0.1.0/26. Verify before continuing.
Spoke VNet
vnet-dbx 10.0.0.64/26 — hosts Databricks worker, container, and Private Endpoint subnets. Non-overlapping with hub.
# 1a. Verify hub VNet exists (created in S2S VPN Guide step 1) az network vnet show -g lakeflow-mssql-on-prem -n vnet-hub --query addressSpace.addressPrefixes -o tsv az network vnet subnet show -g lakeflow-mssql-on-prem --vnet-name vnet-hub -n GatewaySubnet --query addressPrefix -o tsv # 1b. Create the Databricks spoke VNet az network vnet create -g lakeflow-mssql-on-prem -n vnet-dbx --address-prefix 10.0.0.64/26 # 1c. Host-subnet NSG — Databricks injects its own rules at workspace provisioning time az network nsg create -g lakeflow-mssql-on-prem -n nsg-dbx-host # 1d. Container-subnet NSG — separate from host NSG (Databricks requirement) az network nsg create -g lakeflow-mssql-on-prem -n nsg-dbx-container
vnet show (1a)Should list 10.0.0.0/26 and 10.0.1.0/26. Missing? Run S2S VPN Guide step 1 first. vnet-dbx --address-prefix10.0.0.64/26 — sits adjacent to the hub in the same 10.0.0.0/25 parent. Must not overlap hub or on-prem ranges. nsg-dbx-host / nsg-dbx-containerCreate empty — Databricks injects mandatory control-plane rules at workspace provisioning time. Separate NSGs on host and container subnets are required; sharing one NSG between them is not supported.
2

Create Hub → Spoke peering · allows gateway transit to Databricks spoke

--allow-gateway-transit lets the spoke borrow the hub VPN Gateway to reach on-premises. Capture both VNet resource IDs first — --remote-vnet requires the full ARM ID.

--allow-gateway-transit
Hub advertises its VPN Gateway to the spoke. Must be set here before step 3 can use --use-remote-gateways.
--allow-forwarded-traffic
Passes packets that didn't originate in this VNet — required when the hub routes spoke egress.
# Capture VNet resource IDs HUB=$(az network vnet show -g lakeflow-mssql-on-prem -n vnet-hub --query id -o tsv) DBX=$(az network vnet show -g lakeflow-mssql-on-prem -n vnet-dbx --query id -o tsv) # Hub → Spoke peering az network vnet peering create -g lakeflow-mssql-on-prem --vnet-name vnet-hub -n hub-to-dbx \ --remote-vnet $DBX \ --allow-vnet-access \ --allow-forwarded-traffic \ --allow-gateway-transit
⚠ Windows CMD — different syntax for variable capture
for /f %i in ('az network vnet show -g lakeflow-mssql-on-prem -n vnet-hub --query id -o tsv') do set HUB=%i
for /f %i in ('az network vnet show -g lakeflow-mssql-on-prem -n vnet-dbx --query id -o tsv') do set DBX=%i
--query id -o tsvExtracts the full ARM resource ID. --remote-vnet requires the ID (not just the name) when both VNets are in the same resource group — the ID always works. --remote-vnet $DBXFull resource ID of the Databricks spoke VNet. --allow-vnet-accessPermits VM-to-VM and service traffic between the two peered VNets. Required for any connectivity. --allow-gateway-transitAdvertises the hub VPN Gateway to the spoke. Must be active here before step 3's --use-remote-gateways will succeed.
3

Create Spoke → Hub peering · uses remote VPN Gateway for on-prem routing

--use-remote-gateways routes the spoke through the hub VPN Gateway. Requires step 2 to have --allow-gateway-transit already set and the peering to be in Connected state — otherwise this command fails.

--use-remote-gateways
Routes spoke traffic destined for on-prem through the hub's VPN Gateway. Step 2 must complete first.
Ordering matters
This command fails if the hub-to-dbx peering from step 2 is not yet in Connected state with allowGatewayTransit: true.
az network vnet peering create -g lakeflow-mssql-on-prem --vnet-name vnet-dbx -n dbx-to-hub \ --remote-vnet $HUB \ --allow-vnet-access \ --allow-forwarded-traffic \ --use-remote-gateways
--remote-vnet $HUBFull resource ID of the hub VNet — $HUB captured at the start of step 2. --allow-vnet-accessEnables VM-to-VM traffic between spoke and hub. --allow-forwarded-trafficAccepts forwarded packets — required when the hub routes spoke egress through a gateway or NVA. --use-remote-gatewaysUses the hub's VPN Gateway to reach on-prem. Cannot be combined with --allow-gateway-transit on the same peering.
Subnet & NSG Gotchas
⚠️

Reserved subnet names are literal

GatewaySubnet and AzureBastionSubnet must be spelled exactly. Azure looks them up by name — a typo = subnet unusable.

📏

Databricks subnets must be equal size

Host and container subnets require the same prefix length. /26 min, /18 max recommended. Azure reserves 5 IPs per subnet (network, broadcast, 3 Azure).

🔗

Delegations are one-way

Once delegated to Databricks, only Databricks-managed resources can live there. You cannot add a generic VM to a Databricks-delegated subnet.

🛑

Don't attach NSGs to reserved subnets

GatewaySubnet must have no NSG. AzureBastionSubnet can have an NSG but only with strict Microsoft-specified rules.

🔒

NSGs are stateful but directional

Allow inbound does not imply allow outbound. Plan both directions. Priority 100 min for Databricks rules; your deny-all should be > 4000 to let everything else evaluate first.

🧮

Service tags beat CIDR lists

Use AzureDatabricks, Storage, Sql, AzureActiveDirectory service tags instead of IP ranges. Microsoft keeps them current; CIDR lists go stale.
