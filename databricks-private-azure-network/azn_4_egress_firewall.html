User-Defined Routes (UDRs) — Force Tunneling via Azure Firewall

Why attach a custom route table to the Databricks subnets?

By default, a VM in Azure gets a system route that sends 0.0.0.0/0 to Microsoft's backbone (the "Internet" next-hop). That's the public path — fine for ordinary workloads, unacceptable for a locked-down workspace.

A User-Defined Route overrides the default, pointing 0.0.0.0/0 at the private IP of your Azure Firewall instead. Every packet a Databricks worker tries to send to the internet now passes through the firewall where you decide what's allowed.

The route table is attached to the host-subnet and container-subnet. You also add explicit exception routes for things that must not go through the firewall — most importantly the AzureDatabricks service tag, so the SCC connection to the control plane doesn't get black-holed by the firewall.

Azure Firewall Policy — Two Rule Types You Need
Azure Firewall policies have Network rules (IP/port-based, useful for non-HTTP traffic) and Application rules (FQDN/SNI-based, for HTTPS — this is where Databricks FQDNs go). Premium SKU adds TLS inspection & IDPS. Standard is usually enough for Databricks; upgrade to Premium if compliance requires deep packet inspection.
