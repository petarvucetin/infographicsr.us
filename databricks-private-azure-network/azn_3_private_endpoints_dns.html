The Three Databricks Private Endpoints You'll Create
Front-End · UI/API

Workspace databricks_ui_api

Sub-resource: databricks_ui_api

The PE users & tools hit to reach the workspace web UI and REST API. Replaces the public adb-NNN.NN.azuredatabricks.net with a private IP in your workspace-pe-subnet.

Once created and public access is disabled, every UI click and every API call from notebooks, Terraform, CI — even databricks-cli — must traverse this PE.

DNS zone: privatelink.azuredatabricks.net

Browser Auth

Workspace browser_authentication

Sub-resource: browser_authentication

Private path for the interactive SSO/OAuth login flow (login.azuredatabricks.net). Needed so the login popup resolves to your private IP rather than the public endpoint.

Without this, even with UI/API PE in place, browser login can fail with DNS or CORS errors when public access is off.

Same DNS zone: privatelink.azuredatabricks.net

Back-End · SCC

Back-end databricks_ui_api

Sub-resource: databricks_ui_api (same type, diff purpose)

The PE that cluster workers use to reach the control plane over the Secure Cluster Connectivity relay. Required when you disable public IPs on workers (NPIP).

Lives in the pe-subnet inside the Databricks spoke VNet so workers can find it without leaving the VNet.

DNS resolves to the private IP via the same privatelink.azuredatabricks.net zone.

Build It — Storage Account, VNet-Injected Workspace, Private Endpoints, Public Access Disabled
1

Create the ADLS Gen2 storage account · DFS endpoint · replicated SQL data · no public access

Creates the data lake that will hold data replicated from on-premises SQL Server. Hierarchical namespace (HNS) enables the DFS endpoint required by Databricks Delta Lake. Public network access is disabled after the ADLS Private Endpoint is active — run the last command only once PEs for blob and dfs sub-resources are in place (created alongside Databricks PEs in steps 5–7).

Hierarchical namespace
HNS converts Blob Storage into ADLS Gen2 — enables the *.dfs.core.windows.net endpoint, directory-level ACLs, and atomic renames required by Delta Lake.
Name constraint
3–24 lowercase letters and numbers, globally unique across Azure. Replace lakeflowdls with your own name — the example may already be taken.
az storage account create \ -g lakeflow-mssql-on-prem \ -n lakeflowdls \ --sku Standard_LRS \ --kind StorageV2 \ --enable-hierarchical-namespace true \ --min-tls-version TLS1_2 \ --allow-blob-public-access false # Create the filesystem for replicated SQL data az storage fs create \ -n sql-data \ --account-name lakeflowdls \ --auth-mode login # Lock down after ADLS PE is active (steps 5–7) az storage account update \ -g lakeflow-mssql-on-prem \ -n lakeflowdls \ --public-network-access Disabled
⚠ Run the lock-down command only after ADLS Private Endpoints exist
az storage fs create and Databricks workers both call the storage data plane directly. If you run --public-network-access Disabled before the blob + dfs PEs are active, neither the CLI nor the cluster can reach the account.
--enable-hierarchical-namespaceConverts to ADLS Gen2. Enables the DFS endpoint, atomic renames, and POSIX ACLs. Cannot be enabled on an existing account — must be set at creation. --sku Standard_LRSLocally redundant storage. Use Standard_GRS (geo-redundant) or Standard_ZRS (zone-redundant) for production workloads. --kind StorageV2General Purpose v2 — required for hierarchical namespace. BlobStorage and older kinds do not support HNS. --allow-blob-public-access falseDisables anonymous public access to all containers in this account — a baseline security requirement regardless of network access settings. --min-tls-version TLS1_2Rejects connections using TLS 1.0 or 1.1. Required by most compliance frameworks. az storage fs create -n sql-dataCreates the DFS filesystem (equivalent of a container) where replicated SQL data lands. Name matches the migration target path. --public-network-access DisabledRuns as a separate update after PEs are active. Blocks all public internet access — only PE-routed traffic from the VNet is accepted.
2

Create the VNet-injected Databricks workspace · premium SKU · SCC · no public IPs

Creates the workspace wired into host-subnet and container-subnet. --enable-no-public-ip activates Secure Cluster Connectivity — workers get no public IPs and all control-plane traffic routes through the SCC relay PE (step 7). Premium SKU is required for Private Link.

Pre-req: databricks extension
az extension add --name databricks — required once; the az databricks commands are not part of the core CLI.
--enable-no-public-ip
Cannot be toggled off on an existing workspace — re-deployment is required to change it. Always set this from the start for a private workspace.
DBX_VNET_ID=$(az network vnet show -g lakeflow-mssql-on-prem -n vnet-dbx --query id -o tsv) az databricks workspace create \ -g lakeflow-mssql-on-prem \ -n dbx-prod \ --sku premium \ --custom-virtual-network-id $DBX_VNET_ID \ --custom-public-subnet-name host-subnet \ --custom-private-subnet-name container-subnet \ --enable-no-public-ip
⚠ Windows CMD — different syntax for DBX_VNET_ID capture
for /f %i in ('az network vnet show -g lakeflow-mssql-on-prem -n vnet-dbx --query id -o tsv') do set DBX_VNET_ID=%i
Then use %DBX_VNET_ID% in place of $DBX_VNET_ID.
-g / --resource-groupResource group for the workspace — lakeflow-mssql-on-prem. -n / --nameWorkspace name. Azure creates a managed resource group alongside it — name it carefully as it cannot change. --sku premiumPremium tier required for Private Link, customer-managed keys, and IP Access Lists. standard does not support Private Endpoints. --custom-virtual-network-idFull ARM resource ID of vnet-dbx. Captured into $DBX_VNET_ID above. --custom-public-subnet-namehost-subnet — Databricks names this "public" internally (worker VM NICs) regardless of whether NPIP is on. --custom-private-subnet-namecontainer-subnet — carries container network interfaces. Must equal --custom-public-subnet-name prefix length. --enable-no-public-ipBoolean flag — no value needed. Activates SCC; workers route through the relay PE instead of public IPs.
3

Create PE — Databricks UI/API · front-end · workspace-pe-subnet

The PE that users, tools, and notebooks hit for the workspace web UI and REST API. Once public access is disabled in step 9, every connection — including Terraform and databricks-cli — must traverse this endpoint.

Sub-resource
databricks_ui_api — covers workspace REST API and web UI. Distinct from browser_authentication which handles the OAuth redirect hop.
Subnet
workspace-pe-subnet in vnet-dbx — user-facing PEs here; back-end SCC relay PE goes in pe-subnet (step 7).
WS_ID=$(az databricks workspace show -g lakeflow-mssql-on-prem -n dbx-prod --query id -o tsv) az network private-endpoint create \ -g lakeflow-mssql-on-prem -n pe-dbx-ui \ --vnet-name vnet-dbx --subnet workspace-pe-subnet \ --private-connection-resource-id $WS_ID \ --group-id databricks_ui_api \ --connection-name dbx-ui-conn
⚠ Windows CMD — different syntax for WS_ID capture
for /f %i in ('az databricks workspace show -g lakeflow-mssql-on-prem -n dbx-prod --query id -o tsv') do set WS_ID=%i
Then use %WS_ID% in place of $WS_ID in steps 4, 5, and 6.
-g / --resource-groupResource group for the Private Endpoint NIC. --vnet-name / --subnetWhere the PE NIC is injected. Subnet must have --private-endpoint-network-policies Disabled (set in Panel 2, step 8). --private-connection-resource-idFull ARM resource ID of the target workspace — $WS_ID captured above. --group-idPE sub-resource type — databricks_ui_api for UI/API, browser_authentication for OAuth login, same type for SCC relay. --connection-nameName of the private link connection shown in the workspace's Private Endpoint Connections blade.
4

Create PE — browser authentication · OAuth/SSO login · workspace-pe-subnet

Private path for the interactive SSO/OAuth login flow (login.azuredatabricks.net). Without this, browser login fails even when the UI/API PE is healthy — the post-auth redirect still lands on the now-disabled public endpoint.

Sub-resource
browser_authentication — distinct from databricks_ui_api. API clients using PATs don't need this; browser-based users do.
Asymmetry signal
If curl + PAT works but the browser doesn't — that asymmetry means this PE is missing. PAT auth skips the OAuth redirect entirely.
az network private-endpoint create \ -g lakeflow-mssql-on-prem -n pe-dbx-auth \ --vnet-name vnet-dbx --subnet workspace-pe-subnet \ --private-connection-resource-id $WS_ID \ --group-id browser_authentication \ --connection-name dbx-auth-conn
-g / --resource-groupSame resource group as step 5. --private-connection-resource-id$WS_ID — same workspace as step 5, different sub-resource. --group-idbrowser_authentication — routes the login.<region>.azuredatabricks.net OAuth redirect to a private IP. --connection-nameConnection name shown in the workspace's Private Endpoint Connections blade.
5

Create PE — back-end SCC relay · cluster workers · pe-subnet

The PE cluster workers use to reach the control plane via the Secure Cluster Connectivity relay. Required because enableNoPublicIp=true was set in step 2. Lives in pe-subnet so workers can reach it without leaving the spoke VNet.

Sub-resource
databricks_ui_api — same type as step 5 but serving workers, not users. The subnet placement is what distinguishes the two.
Subnet
pe-subnet in vnet-dbx — back-end data-plane PEs (ADLS, KV, SQL, SCC relay) live here, separate from user-facing PEs in workspace-pe-subnet.
az network private-endpoint create \ -g lakeflow-mssql-on-prem -n pe-dbx-scc \ --vnet-name vnet-dbx --subnet pe-subnet \ --private-connection-resource-id $WS_ID \ --group-id databricks_ui_api \ --connection-name dbx-scc-conn
-g / --resource-groupResource group for the PE. -n / --namepe-dbx-scc indicates Secure Cluster Connectivity purpose. --subnetpe-subnet — back-end subnet, distinct from workspace-pe-subnet used for user-facing endpoints. --group-iddatabricks_ui_api — same sub-resource type as step 5; the subnet difference is what routes this to the SCC relay path.
6

Register PE IPs into the DNS zone · zone group · auto-populates A records

The DNS zone group wires the PE's private IP to the private DNS zone automatically. Without this, the zone has no records — DNS queries return nothing and all connections fail regardless of routing.

What it does
Azure auto-creates A records in privatelink.azuredatabricks.net for the PE's private IP. Records stay in sync if the PE IP ever changes.
One zone group per PE
Repeat for pe-dbx-auth and pe-dbx-scc — each PE gets its own zone group pointing to the same zone.
az network private-endpoint dns-zone-group create \ -g lakeflow-mssql-on-prem --endpoint-name pe-dbx-ui -n zone-group-ui \ --private-dns-zone privatelink.azuredatabricks.net \ --zone-name azuredatabricks
-g / --resource-groupResource group of the Private Endpoint (not the DNS zone's RG). --endpoint-namePE to attach the zone group to. Repeat this command with pe-dbx-auth and pe-dbx-scc. -n / --nameZone group name — scoped to the PE, so each PE can have the same group name independently. --private-dns-zoneName or resource ID of the private DNS zone in lakeflow-mssql-on-prem. Must already exist (step 3). --zone-nameFriendly label for this zone entry within the group — used internally by the zone group resource.
9

Disable public network access · THE LOCK · all 3 PEs must be Approved first

The final step that makes the workspace private-only. Once applied, any client not resolving the workspace FQDN to a PE private IP will receive a 403 "privacy settings disallow access" error — including the Azure portal's web terminal.

Pre-condition
All three PE connections must be Approved. Verify: az databricks workspace show -g lakeflow-mssql-on-prem -n dbx-prod --query "properties.privateEndpointConnections[].provisioningState"
--required-nsg-rules
NoAzureDatabricksRules removes the public Databricks control-plane NSG rules — redundant once all traffic routes through the SCC relay PE.
az databricks workspace update \ -g lakeflow-mssql-on-prem -n dbx-prod \ --public-network-access Disabled \ --required-nsg-rules NoAzureDatabricksRules
⚠ Validate connectivity before locking
From your on-prem LAN (or inside the VNet), confirm DNS resolves adb-NNN.azuredatabricks.net to a private IP (nslookup) and TCP 443 is reachable (Test-NetConnection ... -Port 443). Users are locked out during any misconfiguration window after this command runs.
-g / --resource-groupResource group of the workspace. -n / --nameWorkspace name. --public-network-accessDisabled — rejects all connections from public IPs. Only PE-routed traffic is accepted. --required-nsg-rulesNoAzureDatabricksRules — removes Databricks-managed public control-plane NSG rules, now redundant with SCC relay PE active.

All the L3/L4 boxes are checked — peering, routing, PE reachability — yet the browser returns a privacy-restrictions error. The fix is one missing DNS zone link.

The Missing Link

The privatelink.azuredatabricks.net zone had the right A record (10.139.0.4) but was only linked to databricks-vnet — because the PE auto-created that link when its DNS zone group was attached. Azure Private DNS Resolver (which will live in vnet-hub) cannot see a private zone that isn't linked to its own VNet. Without that link, on-prem queries can never resolve to the private IP — no matter how well the rest of the network is wired.

🛠 The Fix — Link the private DNS zone to vnet-hub

① Link the private DNS zone to vnet-hub (the one that was missing)
az network private-dns link vnet create -g lakeflow-mssql-on-prem -n link-hub \ --zone-name privatelink.azuredatabricks.net \ --virtual-network vnet-hub --registration-enabled false
Verify: az network private-dns link vnet list -g lakeflow-mssql-on-prem -z privatelink.azuredatabricks.net -o table now shows two rows — one per VNet — both Succeeded / Completed.
⚠ DNS Resolver + on-prem forwarder are covered in the Site-to-Site VPN Guide (Steps 10 & 11). Step 10 creates the Azure DNS Private Resolver and inbound endpoint in vnet-hub. Step 11 configures the conditional forwarder on your UDM Pro or AdGuard Home, directing *.privatelink.* and *.azuredatabricks.net queries to the inbound endpoint IP over the VPN tunnel.

The Second Missing PE — browser_authentication

Databricks exposes two distinct PE sub-resources for browser-based workspace access. databricks_ui_api handles the workspace UI + REST API. browser_authentication handles the SSO / OAuth redirect flow to login.<region>.azuredatabricks.net. API-only clients (PAT + curl) are fine with just UI/API; browsers need both. Without the second PE, the login redirect still lands on the public endpoint, which is now refusing everyone. Same error message, different hop.

🛠 The Second Fix — Create the browser_authentication PE

① Create the PE and wire it into the same DNS zone
WS_ID=$(az databricks workspace show -g <dbx-rg> -n <workspace-name> --query id -o tsv) az network private-endpoint create -g lakeflow-mssql-on-prem -n pe-dbx-browser-auth \ --vnet-name databricks-vnet --subnet default \ --private-connection-resource-id $WS_ID \ --group-id browser_authentication --connection-name dbx-browser-auth-conn # Register the new PE's IP into the same private DNS zone az network private-endpoint dns-zone-group create -g lakeflow-mssql-on-prem \ --endpoint-name pe-dbx-browser-auth -n zone-group-auth \ --private-dns-zone privatelink.azuredatabricks.net --zone-name azuredatabricks
Verify: az databricks workspace show --query "properties.privateEndpointConnections[].groupIds" now returns both ["databricks_ui_api","browser_authentication"], both Approved.
② Flush DNS caches & retest both hops
# AdGuard Home: Settings → DNS → Clear cache # Windows client: ipconfig /flushdns # Both hops must now resolve privately: nslookup adb-NNN...azuredatabricks.net # → 10.139.0.4 (UI/API PE) nslookup login.NN.azuredatabricks.net # → 10.139.0.5 (browser-auth PE) # Open workspace in browser — full OAuth handshake now stays private
Final state: Browser loads workspace URL → Entra ID sign-in page → lands on workspace home. Both hops traverse PEs; nothing touches the public internet. Truly private-only.
⚠ Diagnostic shortcut: if curl -H "Authorization: Bearer <PAT>" https://adb-NNN.../api/2.0/workspace/list works but the browser doesn't — that asymmetry alone is a strong signal that browser_authentication is the missing piece. PAT auth skips the OAuth redirect hop entirely.
