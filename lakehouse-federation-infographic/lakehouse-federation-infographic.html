🔍 What Is Lakehouse Federation

Query SQL Server Without Moving It

Lakehouse Federation lets Unity Catalog create a live read connection to external databases. SQL Server appears as a foreign catalog — you query it with regular SQL alongside your Delta tables. No replication pipeline, no copy of data.

🔌
How it worksDatabricks pushes your SQL query down to SQL Server via JDBC. Results stream back and are never persisted in Databricks storage unless you explicitly write them.
🔐
Who controls itUnity Catalog manages the connection credential, access policies, and audit logs — same governance plane as your Delta tables.
📍
Supported sourcesSQL Server, MySQL, PostgreSQL, Snowflake, Azure SQL, Redshift, BigQuery (and growing).

🌐 Network Path Options (Pick One)
🔒
Azure ExpressRoute
Best for production workloads
Dedicated private circuit from your datacenter to Azure. Databricks cluster (VNet-injected) reaches SQL Server over ExpressRoute — never touches public internet.
Typical latency: 1–5 ms · Bandwidth: up to 100 Gbps
ProLowest latency, highest bandwidth, fully private
ConCost and lead time to provision (weeks)
🛡️
Site-to-Site VPN
Faster to set up, widely used
IPSec VPN tunnel between your on-prem router and Azure VNet Gateway. Databricks clusters in VNet injection reach SQL Server through the tunnel.
Typical latency: 10–50 ms · Bandwidth: up to 10 Gbps
ProHours to set up, low cost, encrypted
ConHigher latency, internet-routed
🌍
Public IP + Firewall Rules
Dev/POC only — not for prod
SQL Server exposed on public internet with IP allow-listing for Databricks NAT IPs. Requires SSL/TLS. Quick for proofs of concept only.
Typical latency: 20–100 ms · Variable bandwidth
ProFastest to configure for a POC
ConSecurity risk — never use in prod
⚙️ Setup: Four SQL Statements
1

Create the Connection

Runs once · Requires CREATE CONNECTION privilege
CREATE CONNECTION onprem_sqlserver TYPE sqlserver OPTIONS ( host '10.0.0.5', -- internal IP port '1433', user 'svc_databricks', -- read-only acct password secret( 'my-scope', 'sql-pass' ) );
Tip: Store credentials in Databricks Secrets, never in plain text. Use a read-only SQL login.
2

Create the Foreign Catalog

Maps SQL Server DB → Unity Catalog namespace
CREATE FOREIGN CATALOG sqlserver_prod USING CONNECTION onprem_sqlserver OPTIONS ( database 'ProductionDB' -- SQL Server DB name );
Result: sqlserver_prod now appears in Unity Catalog Explorer. Schemas and tables are auto-discovered.
3

Query Live (Federated Read)

No data movement — SQL pushed to SQL Server
-- Join federated SQL Server with Delta table SELECT o.order_id, o.amount, c.segment -- from Delta table FROM sqlserver_prod.dbo.orders o JOIN main.gold.customers c ON o.customer_id = c.id WHERE o.order_date >= '2024-01-01'
Pushdown: The WHERE clause is sent to SQL Server — only filtered rows come back over JDBC.
4

Land Into Delta (Optional)

Copy federated data into lakehouse storage
-- One-time full load or incremental pattern CREATE OR REPLACE TABLE main.bronze.orders AS SELECT *, current_timestamp() AS ingested_at FROM sqlserver_prod.dbo.orders WHERE modified_date > :last_watermark; -- Or use Delta Live Tables for automation
Best for: Heavy analytics, ML training, or when you need offline access / time travel.

📥 Patterns for Landing SQL Server Data in the Lakehouse
1️⃣

Full Load → Bronze (CTAS)

Simple · Best for small/medium tables

Run a CREATE TABLE AS SELECT from the federated catalog. Overwrites Bronze table on each run. Simple, no state management.

CREATE OR REPLACE TABLE main.bronze.orders AS SELECT *, now() AS _loaded_at FROM sqlserver_prod.dbo.orders;
2️⃣

Watermark Incremental Load

Efficient · Needs a modified_date column

Track the last loaded timestamp. Each run only pulls rows newer than the watermark and merges into Delta.

MERGE INTO main.bronze.orders t USING ( SELECT * FROM sqlserver_prod.dbo.orders WHERE modified_date > :last_watermark ) s ON t.id = s.id WHEN MATCHED THEN UPDATE SET * WHEN NOT MATCHED THEN INSERT *;
3️⃣

Delta Live Tables Pipeline

Automated · Declarative · Monitored

Define a DLT batch table sourced from the foreign catalog. Databricks handles scheduling, retries, and data quality expectations automatically.

@dlt.table def orders_bronze(): return spark.read.table( "sqlserver_prod.dbo.orders" ).withColumn("_loaded_at", now())
4️⃣

Medallion Architecture Flow

Full lakehouse pattern · Bronze → Silver → Gold

Federation is the Bronze entry point. Silver applies cleansing. Gold serves aggregated metrics to BI tools — all governed by Unity Catalog.

-- Bronze: raw from SQL Server sqlserver_prod.dbo.orders → bronze.orders -- Silver: cleaned, typed, deduplicated bronze.orders → silver.orders_clean -- Gold: business-ready aggregates silver.* → gold.revenue_by_region
⚠️ Known Limitations to Plan Around
📖

Read-Only

Federation is read-only. You cannot INSERT, UPDATE, or DELETE back to SQL Server through the foreign catalog.

🐌

Performance at Scale

Scanning 100M+ row tables federated is slow. Network + SQL Server I/O becomes the bottleneck. Replicate large tables instead.

📡

Always-On Network Required

Queries fail if the VPN/ExpressRoute is down. No caching layer. Build fallback logic for production workloads.

🔧

Partial Pushdown

Not all SQL expressions push down. Complex Spark functions run client-side in Databricks, which can pull more data than expected.

🎯 Pro Tips
🔑

Always Use a Read-Only Service Account

Create a dedicated SQL login with SELECT-only on required schemas. Store the password in Databricks Secrets — never hardcode credentials in notebooks or SQL.

⚡

Push Filters Early

Always filter on indexed columns (date ranges, IDs) before joining or aggregating. This maximises pushdown to SQL Server and minimises data transferred over JDBC.

🏗️

Use Federation as Your Bronze Entry Point

Land data into a Bronze Delta table first, then apply transformations in Silver/Gold. This decouples your analytics from the live SQL Server dependency.

📊

Monitor with Unity Catalog Audit Logs

All queries against the foreign catalog are logged in the Unity Catalog system tables (system.access.audit). Use this for compliance and query profiling.

🔀

Combine Federation + Replication

Use federation for small/lookup tables (live) and Partner Connect replication for large fact tables (Delta). Mix strategies in the same lakehouse — Unity Catalog unifies both.

🧪

Test Your Network Path First

Before creating the connection, verify JDBC reachability from a Databricks cluster: spark.read.jdbc(url, "sys.tables", ...). Saves debugging time later.
