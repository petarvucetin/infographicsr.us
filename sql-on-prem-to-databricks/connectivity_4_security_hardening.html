Defence-in-Depth Security Model
1

🔶 Network Layer

  • All on-prem ↔ Azure traffic over private ExpressRoute or VPN — zero public internet exposure
  • Databricks VNet isolated via NSG — deny all, allow only required service tags
  • NAT Gateway for stable, predictable outbound IPs (post-March 2026 requirement)
  • Optional Azure Firewall in Transit VNet for deep packet inspection and egress control
2

🔷 Identity Layer

  • Dedicated SQL Server service account — read-only, CDC access only, no sysadmin
  • Credentials stored in Unity Catalog Connection object — encrypted, never in code
  • Azure Managed Identity for ADF / Databricks to access ADLS Gen2 — no stored secrets
  • SQL Server auth via UC Connection: username/password or Entra ID OAuth
3

🟢 Data Layer

  • TLS 1.2+ enforced on all JDBC connections (SQL Server → Databricks)
  • ADLS Gen2 private endpoint — storage traffic never leaves Azure backbone
  • Unity Catalog column masking on PII fields — masked at Silver, inherited to Gold
  • Delta Lake transaction log provides immutable audit trail of all ingested rows
4

🟣 Governance Layer

  • Unity Catalog RBAC — table, schema, column-level access enforced centrally
  • Audit log: all table access, pipeline runs, and credential usage captured in system tables
  • Databricks workspace IP access list — restrict workspace UI access to corporate IPs only
  • Rotate SQL Server service account passwords every 90 days — update UC Connection object

🔒 Secure Cluster Connectivity (SCC) — Enable This

SCC removes all public IPs from Databricks cluster nodes. Cluster nodes have only private IPs. All control plane communication goes through a secure tunnel outbound — no inbound ports opened on the cluster NSG.

✅ What SCC Eliminates

  • No public IPs on any cluster node
  • Removes inbound NSG rules for ports 22 and 5557
  • No inbound connections from Databricks control plane
  • Clusters communicate to control plane via SCC relay outbound only
  • Required for most enterprise security postures

⚙️ How to Enable SCC

  • Azure Portal: Create Databricks workspace → Networking → Enable Secure Cluster Connectivity: YES
  • Add UDR for SCC Relay IP (not control plane NAT IP) in your route table
  • Verify: cluster node IPs are in private subnet range — no public IPs in managed resource group
  • SCC + VNet injection = strongest security posture
Azure Private Link Options (Optional but Recommended)
Inbound — Front-End

Workspace Private Endpoint

Creates a private endpoint in your VNet for the Databricks workspace UI and REST API. Users connect to the workspace over the private endpoint — no public HTTPS access needed. Ideal for corporate users connecting over ExpressRoute/VPN.

Back-End — Compute Plane

Classic Compute Plane Private Link

Secures the connection from Databricks cluster nodes to the control plane services (secure cluster connectivity relay). A private endpoint is deployed in the workspace VNet for the SCC relay service — control plane traffic stays fully private.

Serverless — Lakeflow Connect

Serverless Outbound Private Connectivity

Enables Lakeflow Connect (serverless compute) to reach your on-prem SQL Server via private networking. Configure via the Databricks workspace serverless egress controls — ensures the Ingestion Gateway uses the private path to SQL Server.

🌐 Custom DNS Configuration (Optional)

When to Use Custom DNS
When your org has on-prem DNS servers that need to resolve both Azure private endpoints and on-prem SQL Server hostnames from within Databricks clusters.
Required DNS Forwarding
Custom DNS servers must forward Azure artifact resolution requests to the Azure recursive resolver (168.63.129.16) — critical for Databricks control plane connectivity.
Private DNS Zones
If using Private Link for workspace or ADLS, link the Private DNS Zones (privatelink.azuredatabricks.net, privatelink.dfs.core.windows.net) to the Databricks VNet and transit VNet.
Azure DNS Private Resolver
For users connecting over ExpressRoute/VPN who need to resolve Databricks private endpoints, deploy Azure DNS Private Resolver in the hub VNet to handle split-horizon DNS.
SQL Server Hostname
Databricks connects to SQL Server via hostname in UC Connection. Ensure the hostname resolves to the on-prem private IP from within the Databricks VNet via the custom DNS or hosts file on the SHIR (if using ADF).
Verify DNS Resolution
From a Databricks notebook: %sh nslookup <sql-server-hostname> — should return the on-prem private IP, not a public IP.
