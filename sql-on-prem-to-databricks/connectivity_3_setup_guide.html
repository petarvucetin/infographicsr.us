Phase 1 · On-Premises
Prepare SQL Server & On-Prem Network
a
Enable Change Tracking or CDC
Enable CT (tables with PKs) or CDC (tables without PKs) on the target database and tables. SQL Server 2012+ required.
b
Create Dedicated Service Account
Create a dedicated SQL login for Databricks with minimum permissions: db_datareader + CDC read privileges only.
c
Open Firewall Port 1433
Allow inbound TCP 1433 from the Azure Virtual Network Gateway subnet IP range on-premises firewall and Windows Firewall.
⚠ SQL Server Agent must be running for CDC capture jobs. Verify with: SELECT status_desc FROM sys.dm_server_services WHERE servicename LIKE '%Agent%'
2
Phase 2 · Azure Networking
Set Up Transit VNet with Virtual Network Gateway
a
Create or Identify Transit VNet (Hub)
This is a separate, pre-existing VNet — NOT the Databricks VNet. Must include a GatewaySubnet (/27 or larger) for the VNet Gateway.
b
Deploy Azure Virtual Network Gateway
Choose type: ExpressRoute (for dedicated circuit) or VPN Gateway (for site-to-site VPN). SKU: VpnGw1 or higher for production.
Portal: Create resource → Networking → Virtual Network Gateway Gateway type: ExpressRoute or VPN SKU: VpnGw1 (min for production) Virtual network: [your transit VNet] Public IP: create new static
📌 If ExpressRoute already exists between on-prem and Azure, configure the gateway per the ExpressRoute VNet gateway guide — do not create a new circuit.
3
Phase 2 · Azure Networking
Deploy Databricks with VNet Injection
a
Create Databricks Workspace in Your VNet
In Azure Portal → Create Databricks → Networking tab → select your VNet. Configure two dedicated subnets (host + container, /26 minimum each).
b
Attach NAT Gateway (New VNets post-March 2026)
Attach a NAT Gateway to both host and container subnets to provide stable outbound connectivity. Required for new VNets created after March 31, 2026.
Portal: Azure Databricks → Create Networking tab: Virtual network: [Databricks VNet] Host subnet: 10.x.x.x/26 Container subnet: 10.x.y.x/26 Enable Secure Cluster Connectivity: YES NAT Gateway: attach to both subnets
⚠ Subnet CIDR ranges cannot be changed after deployment. Plan carefully and size generously (prefer /24 over /26 for production).
4
Phase 3 · Databricks VNet
Peer Databricks VNet to Transit VNet
a
Create VNet Peering (Databricks Side)
Navigate to Databricks VNet → Peerings → Add. Set remote VNet to Transit VNet. Enable "Use Remote Gateways".
b
Create VNet Peering (Transit Side)
Navigate to Transit VNet → Peerings → Add. Set remote VNet to Databricks VNet. Enable "Allow Gateway Transit".
Databricks VNet peering settings: ✅ Use Remote Gateways: ON ✅ Allow Forwarded Traffic: ON (if needed) ❌ Allow Gateway Transit: OFF Transit VNet peering settings: ✅ Allow Gateway Transit: ON ✅ Allow Forwarded Traffic: ON (if needed) ❌ Use Remote Gateways: OFF
✅ After peering, cluster nodes in the Databricks VNet will automatically route on-prem-bound traffic through the Transit VNet gateway.
5
Phase 4 · Routing
Create User-Defined Routes (UDRs)
a
Create Route Table with BGP Propagation ON
Azure portal → Route tables → Create. Enable "BGP route propagation". Associate with BOTH Databricks subnets (host and container).
b
Add Required UDRs for Control Plane
Add routes for SCC Relay IP, Webapp IP, Metastore IP, Artifact Blob, Log Blob, ADLS, and EventHubs — all with Next hop: Internet.
Routes to add (Next hop = Internet): • SCC Relay IP (if SCC enabled) • Control plane NAT IP (if SCC disabled) • Webapp IP • Metastore IP • Artifact Blob storage IP • Log Blob storage IP • Workspace ADLS storage IP • Event Hubs IP Get IPs from: Azure Databricks UDR docs
⚠ Without these UDRs, after peering Databricks clusters cannot communicate with the Databricks control plane and cluster creation will fail.
6
Phase 5 · Validate
Test Connectivity End-to-End
a
Create a Test Cluster
Launch Workspace → Compute → Create Cluster. If cluster fails, review UDRs first — missing control plane routes are the most common cause.
b
Ping On-Prem IP from Notebook
Open a notebook and ping an on-prem IP to verify the private path is working. Then test SQL Server port access.
-- In Databricks notebook: %sh ping <on-prem-server-ip> # Test SQL Server port 1433 nc -zv <sql-server-ip> 1433 -- Test Lakeflow Connect SELECT MAX(_commit_timestamp) AS last_updated FROM main.sprouts_silver.orders;
✅ Successful ping + nc -zv confirms the private path. If nc fails but ping works, check NSG rules and on-prem firewall for port 1433.

✅ Validation Checklist

  • Cluster creates successfully in the VNet-injected workspace
  • Cluster node IPs are in the expected host subnet range (check Spark UI → Executors)
  • ping <on-prem-ip> succeeds from Databricks notebook
  • nc -zv <sql-ip> 1433 returns "Connection succeeded"
  • Lakeflow Connect Gateway starts and writes to staging volume without errors
  • Ingestion Pipeline completes first run — check destination table row counts match source

🔧 Common Failure Points

  • Cluster creation fails: Missing UDRs for SCC Relay or Webapp IP — check route table is associated to both subnets
  • Ping fails: VNet peering misconfigured — verify "Use Remote Gateways" and "Allow Gateway Transit" settings
  • Port 1433 blocked: On-prem or Azure NSG blocking TCP 1433 — check both sides
  • Lakeflow Gateway fails: Staging catalog is a foreign catalog — must be native UC catalog
  • Outbound fails (new VNet): Missing NAT Gateway after March 31, 2026 policy change
