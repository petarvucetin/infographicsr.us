🌐 Databricks VNet Layout — What to Deploy
Subnet 1 — Required
Host Subnet
e.g. 10.179.0.0/26 (minimum /26)
  • Also called the "public subnet" (even with SCC, IPs remain private)
  • Databricks cluster node host VMs attach network interfaces here
  • Each cluster node uses 1 IP in this subnet
  • Cannot share with other Azure services or workspaces
  • Must be delegated to Microsoft.Databricks/workspaces
Subnet 2 — Required
Container Subnet
e.g. 10.179.1.0/26 (minimum /26)
  • Also called the "private subnet"
  • Databricks container runtime network interfaces attach here
  • Each cluster node uses 1 IP in this subnet (total: 2 IPs per node)
  • Cannot share with other Azure services or workspaces
  • Must be delegated to Microsoft.Databricks/workspaces
Post-March 2026 — Required for new VNets
NAT Gateway
Attached to both subnets
  • After March 31, 2026 new VNets have NO default outbound internet access
  • NAT Gateway provides stable, predictable egress public IPs
  • Enables Databricks control plane communication (SCC relay, metastore)
  • Stable IPs can be allow-listed on corporate firewalls and SaaS tools
  • Existing workspaces / VNets are NOT affected by this change

🔗 VNet Peering Configuration (Databricks VNet ↔ Transit VNet)

On Databricks VNet Side
  • Enable "Use Remote Gateways" — routes on-prem traffic via transit gateway
  • Enable "Allow Forwarded Traffic" if experiencing routing issues
  • Do NOT enable "Allow Gateway Transit" on this side
⟷
On Transit VNet Side
  • Enable "Allow Gateway Transit" — shares the VPN/ExpressRoute gateway
  • Enable "Allow Forwarded Traffic" on both sides if needed
  • Transit VNet must already have a VNet Gateway deployed
🏷️
Subnet Delegation — Automatic NSG Management
When you deploy Databricks via VNet injection, both subnets are automatically delegated to Microsoft.Databricks/workspaces. This delegation allows Azure Databricks to auto-configure and manage all required NSG rules. Never manually delete or modify these auto-managed rules — doing so will break cluster creation. Add your own custom rules (e.g. deny cross-workspace traffic) in a separate NSG associated to other subnets in the same or peered VNets.
🗺️
User-Defined Routes (UDRs) — Required After Peering
After peering Databricks VNet to Transit VNet, Azure auto-configures most routes — but the return route from cluster nodes to the Databricks control plane is not created automatically. You must manually add UDRs for: SCC Relay IP, Webapp IP, Metastore IP, Artifact Blob, Log Blob, ADLS, and Event Hubs IPs. Set Next hop type to "Internet" for these routes. Associate the route table with both the host and container subnets. Enable BGP route propagation on the route table. Find the IP addresses via the Azure Databricks UDR reference documentation.
