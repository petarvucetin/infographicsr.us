Concept — What a Private Endpoint Actually Is

Azure PaaS service public DNS by default

  • Databricks workspace · adb-123.12.azuredatabricks.net
  • ADLS Gen2 · myacct.dfs.core.windows.net
  • Key Vault · myvault.vault.azure.net
  • Azure SQL · mysrv.database.windows.net
  • These hostnames resolve to public IPs on the internet by default
PRIVATE LINK 🔗 Private Endpoint

Private Endpoint NIC in your VNet

  • A network interface lives in your subnet with a private IP (e.g. 10.139.128.7)
  • Azure injects a CNAME / A-record pointing the public FQDN at the private IP — via a Private DNS zone
  • All traffic from the VNet (and peered VNets) to that service now goes over the Azure backbone — never the internet
  • Once public access is disabled on the service, there is no other path

DNS Resolution Flow — From On-Prem to Private Endpoint

How your laptop (or an on-prem service) resolves adb-123.12.azuredatabricks.net to a private Azure IP

1
On-prem query

Laptop asks its local DNS (e.g. UDM Pro's resolver) for adb-123.12.azuredatabricks.net.

2
Conditional forwarder

UDM / on-prem DNS has a forwarder rule: "for *.azuredatabricks.net, ask Azure DNS Resolver inbound endpoint."

3
Azure Private DNS Resolver

Inbound endpoint receives the query over the S2S VPN tunnel, forwards to Azure-provided DNS (168.63.129.16).

4
Private DNS zone match

Azure-provided DNS matches the CNAME to *.privatelink.azuredatabricks.net, returns the Private Endpoint's IP (e.g. 10.139.129.5).

5
Traffic over VPN

Laptop connects to 10.139.129.5 — reachable via the S2S VPN + VNet peering. Request never touches the public internet.

Build It — Create Zones & Link to Both VNets
1

Create the private DNS zones · once per service type · hub resource group

Each Azure PaaS service type needs its own private DNS zone. Centralise all zones in lakeflow-mssql-on-prem so they can be linked to both VNets from one place. Create all four before creating any Private Endpoints.

Where zones live
Hub resource group lakeflow-mssql-on-prem — centralising here avoids duplication and simplifies VNet linking across hub and Databricks spoke.
Four required zones
privatelink.azuredatabricks.net, …blob…, …dfs…, …vaultcore.azure.net
az network private-dns zone create -g lakeflow-mssql-on-prem -n privatelink.azuredatabricks.net az network private-dns zone create -g lakeflow-mssql-on-prem -n privatelink.blob.core.windows.net az network private-dns zone create -g lakeflow-mssql-on-prem -n privatelink.dfs.core.windows.net az network private-dns zone create -g lakeflow-mssql-on-prem -n privatelink.vaultcore.azure.net
-g / --resource-groupHub resource group — zones are shared infrastructure, not per-workspace. Keeping them in the hub RG avoids scattered lifecycle management. -n / --nameZone name must exactly match the Azure service's private DNS suffix (see table above). A typo here means PEs register A records in the wrong zone and queries never resolve.
2

Link every DNS zone to both VNets · hub + Databricks spoke · registration disabled

A zone that exists but isn't linked to a VNet is invisible to resources in that VNet. Link each zone to both vnet-hub and vnet-dbx so the DNS Resolver inbound endpoint, cluster workers, and management VMs all resolve the same private IPs.

Why link to hub too?
vnet-hub hosts the Private DNS Resolver inbound endpoint. Without a zone link here, on-prem queries forwarded through the resolver never resolve to private IPs — regardless of how well the rest of the network is wired.
--registration-enabled false
Prevents Azure auto-registering VM hostnames in the zone. You want explicit PE A records only — auto-registration pollutes the zone and causes resolution surprises.
# --- privatelink.azuredatabricks.net --- az network private-dns link vnet create \ -g lakeflow-mssql-on-prem -n dbx-zone-hub --zone-name privatelink.azuredatabricks.net \ --virtual-network vnet-hub --registration-enabled false az network private-dns link vnet create \ -g lakeflow-mssql-on-prem -n dbx-zone-dbx --zone-name privatelink.azuredatabricks.net \ --virtual-network vnet-dbx --registration-enabled false # --- privatelink.blob.core.windows.net --- az network private-dns link vnet create \ -g lakeflow-mssql-on-prem -n blob-zone-hub --zone-name privatelink.blob.core.windows.net \ --virtual-network vnet-hub --registration-enabled false az network private-dns link vnet create \ -g lakeflow-mssql-on-prem -n blob-zone-dbx --zone-name privatelink.blob.core.windows.net \ --virtual-network vnet-dbx --registration-enabled false # --- privatelink.dfs.core.windows.net --- az network private-dns link vnet create \ -g lakeflow-mssql-on-prem -n dfs-zone-hub --zone-name privatelink.dfs.core.windows.net \ --virtual-network vnet-hub --registration-enabled false az network private-dns link vnet create \ -g lakeflow-mssql-on-prem -n dfs-zone-dbx --zone-name privatelink.dfs.core.windows.net \ --virtual-network vnet-dbx --registration-enabled false # --- privatelink.vaultcore.azure.net --- az network private-dns link vnet create \ -g lakeflow-mssql-on-prem -n kv-zone-hub --zone-name privatelink.vaultcore.azure.net \ --virtual-network vnet-hub --registration-enabled false az network private-dns link vnet create \ -g lakeflow-mssql-on-prem -n kv-zone-dbx --zone-name privatelink.vaultcore.azure.net \ --virtual-network vnet-dbx --registration-enabled false
-g / --resource-groupResource group containing the DNS zone — lakeflow-mssql-on-prem. -n / --nameLink name convention: {zone-prefix}-zone-{vnet-suffix} e.g. dbx-zone-hub, blob-zone-dbx. Name is local metadata only. --zone-nameExact zone name as created in step 1. Must match character-for-character. --virtual-networkVNet to link. Each zone needs two link commands — one for vnet-hub (so the DNS resolver can see it), one for vnet-dbx (so cluster workers can resolve). --registration-enabled falseMust be false. Linking with true auto-adds VM A records, polluting the PE records and causing resolution surprises.
