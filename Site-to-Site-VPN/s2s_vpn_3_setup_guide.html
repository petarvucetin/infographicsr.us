Before You Start — Prerequisites Checklist
📋

Non-Overlapping CIDRs

Your on-prem space (e.g. 192.168.0.0/24) and Azure VNet (e.g. 10.0.0.0/26) must not overlap — not even partially.

🌐

Static Public IP on-prem

VPN device needs a routable, static public IPv4. Dynamic IP works only with FQDN on some SKUs — avoid for production.

🔥

Firewall Ports Open

Allow outbound UDP 500 & UDP 4500 and IP protocol 50 (ESP). Same on any upstream ISP/edge firewall.

🛠️

Compatible VPN Device

Check Microsoft's validated-devices list. Prefer IKEv2, route-based. Have admin access & latest firmware.

The 11-Step Build — Azure Side, Then On-Prem
1

Create the hub Virtual Network and all required subnets · Azure Portal or CLI

Create vnet-hub with two address prefixes — one for the VPN gateway and workload subnet, one for the DNS Private Resolver subnets. Carve out all four subnets up front before provisioning any resources into them.

Address space 1
10.0.0.0/26 — VPN GatewaySubnet (/27) + workload subnet (/27)
Address space 2
10.0.1.0/26 — DNS resolver inbound (/28) + outbound (/28)
# 1a. Create VNet with both address prefixes — no inline subnet az network vnet create -g lakeflow-mssql-on-prem -n vnet-hub \ --address-prefixes 10.0.0.0/26 10.0.1.0/26 # 1b. GatewaySubnet — name must be exactly this; Azure looks it up by name az network vnet subnet create -g lakeflow-mssql-on-prem --vnet-name vnet-hub \ -n GatewaySubnet --address-prefix 10.0.0.32/27 # 1c. Workload subnet for VMs and private endpoints az network vnet subnet create -g lakeflow-mssql-on-prem --vnet-name vnet-hub \ -n snet-app --address-prefix 10.0.0.0/27 # 1d. DNS resolver inbound subnet (delegated automatically on endpoint creation in step 10) az network vnet subnet create -g lakeflow-mssql-on-prem --vnet-name vnet-hub \ -n dns-resolver-in --address-prefix 10.0.1.0/28 # 1e. DNS resolver outbound subnet az network vnet subnet create -g lakeflow-mssql-on-prem --vnet-name vnet-hub \ -n dns-resolver-out --address-prefix 10.0.1.16/28
--address-prefixesSpace-separated CIDRs — a single VNet can span multiple non-contiguous blocks. Must not overlap on-prem or peered VNets. GatewaySubnet /27Azure VPN Gateway requires this exact name. Do not attach an NSG or a UDR that overrides 0.0.0.0/0. Minimum /27; /26 gives headroom for Route Server coexistence. snet-app /27Workload subnet — hosts VMs, private endpoints, or other resources reachable from on-prem over the VPN tunnel. dns-resolver-in /28Inbound resolver subnet. Azure delegates it to Microsoft.Network/dnsResolvers automatically when the inbound endpoint is provisioned in step 10. dns-resolver-out /28Outbound resolver subnet. Non-overlapping /28 within the same 10.0.1.0/26 prefix — used when Azure forwards queries to on-prem DNS.
2

Add the GatewaySubnet · /27 minimum, /26 recommended

A dedicated subnet named exactly GatewaySubnet. Do not attach an NSG. Do not add a UDR that overrides 0.0.0.0/0.

Why /26?
Leaves room for future Route Server, ExpressRoute gateway coexistence, or larger SKUs that need more IPs.
Example prefix
10.0.0.32/27 (upper half of VNet)
az network vnet subnet create -g lakeflow-mssql-on-prem --vnet-name vnet-hub -n GatewaySubnet --address-prefix 10.0.0.32/27
-g / --resource-groupResource group of the parent VNet. --vnet-nameVNet to add this subnet to. -n / --nameSubnet name. For the VPN gateway subnet it must be exactly GatewaySubnet — Azure looks it up by this name. --address-prefixSubnet CIDR inside the VNet's address space. /27 min, /26 recommended.

Create a Public IP for the VPN Gateway · Basic SKU · Dynamic

The Basic VPN Gateway SKU requires a Basic-SKU Dynamic public IP. Standard/Static IPs are only needed for VpnGw1+ SKUs.

az network public-ip create -g lakeflow-mssql-on-prem -n pip-vpngw-01 --sku Basic --allocation-method Dynamic

4

Create the VPN Gateway · type=Vpn, vpnType=PolicyBased — takes 20–25 min

The Basic SKU is the lowest-cost option — suitable for dev/test and low-throughput S2S. No BGP, no active-active, no zone redundancy. This step is the slow one — Azure provisions the gateway VM behind the scenes.

SKU
Basic — ~100 Mbps, up to 10 S2S tunnels, PolicyBased or RouteBased
Active-active?
Not supported on Basic. Upgrade to VpnGw1+ for HA / active-active
az network vnet-gateway create -g lakeflow-mssql-on-prem -n vpngw-hub \ --vnet vnet-hub --public-ip-addresses pip-vpngw-01 \ --gateway-type Vpn --vpn-type PolicyBased --sku Basic --no-wait
-g / --resource-groupResource group for the VPN Gateway. -n / --nameVPN Gateway resource name. --vnetVNet the gateway lives in. It attaches automatically to that VNet's GatewaySubnet. --public-ip-addressesThe Basic-SKU Dynamic public IP created in step 3. --gateway-typeVpn for Site-to-Site / P2S. Alternatives: ExpressRoute, LocalGateway. --vpn-typePolicyBased (IKEv1, static routing — Basic only) or RouteBased (IKEv2, also supported on Basic but with limited features). --skuBasic — lowest cost tier. Upgrade to VpnGw1–VpnGw5 (append AZ for zone-redundancy) for production workloads. --no-waitReturn immediately instead of polling. Provisioning runs for 20–25 min in the background.
5

Create the Local Network Gateway · Azure's representation of your on-prem

This resource is just metadata: on-prem public IP + the CIDRs reachable behind it (or BGP peer IP/ASN if you're using BGP).

⚠ Replace both placeholder values before running this command
  1. 198.51.100.10 → your on-prem VPN device's actual public WAN IP. Find it on your router's WAN status page, or run curl ifconfig.me from any on-prem host. Using the wrong IP here is the most common reason the tunnel gets stuck — Azure will silently drop all IKE packets from an unrecognised source.
  2. 192.168.0.0/24 → the specific LAN subnet(s) you want to route through the tunnel (e.g. 192.168.1.0/24). Only list subnets that need access to Azure. To update later: az network local-gateway update -g <rg> -n lng-onprem --local-address-prefixes <CIDRs>
az network local-gateway create -g lakeflow-mssql-on-prem -n lng-onprem \ --gateway-ip-address 198.51.100.10 \ --local-address-prefixes 192.168.0.0/24
-g / --resource-groupResource group for the Local Network Gateway. -n / --nameLNG resource name. This is just metadata that describes your on-prem side. --gateway-ip-address⚠ Placeholder — replace 198.51.100.10 with your VPN device's real public WAN IP. Azure uses this to identify your device; a wrong value causes it to silently reject all incoming IKE packets. --local-address-prefixes⚠ Placeholder — replace 192.168.0.0/24 with your actual LAN subnet(s). Space-separated for multiple. Must match what your on-prem device advertises as its local traffic selector.
6

Create the Connection + Shared Key · Ties gateway + LNG together

Generate a strong PSK (32+ random chars). Store it in Key Vault. Use the same key exactly on the on-prem device. Optionally set an IPsec/IKE policy to force modern crypto.

# Generate a PSK openssl rand -base64 32 # Create connection az network vpn-connection create -g lakeflow-mssql-on-prem -n cn-onprem \ --vnet-gateway1 vpngw-hub --local-gateway2 lng-onprem \ --shared-key "<PSK>" # Add --enable-bgp (no value) if you want dynamic routing
-g / --resource-groupResource group for the Connection resource. -n / --nameConnection resource name. One connection = one tunnel pair. --vnet-gateway1Azure-side VPN Gateway (side 1 of the tunnel). --local-gateway2On-prem Local Network Gateway (side 2). VNetGw + LNG pairing = S2S/IPsec connection type. --shared-keyPre-shared key (PSK). 32+ random chars. Must match on-prem device exactly. Store in Key Vault. --enable-bgpFlag only — include --enable-bgp (no value) to turn on dynamic BGP route exchange. Omit it for static routing (the default). Passing --enable-bgp false fails with "unrecognized arguments: false".

8

Create the DNS Resolver inbound subnet · receives on-prem privatelink queries

The inbound endpoint receives privatelink.* queries forwarded from your on-premises device (e.g. UDM Pro), resolving them to private IPs so on-prem clients reach Azure PaaS services over the VPN tunnel without hitting public endpoints.

Purpose
Accepts DNS queries forwarded from on-prem. Azure-provided DNS resolves them against Private DNS zones and returns the PE's private IP.
Delegation
Delegated to Microsoft.Network/dnsResolvers automatically when the inbound endpoint is provisioned — no manual delegation needed here.
az network vnet subnet create -g lakeflow-mssql-on-prem --vnet-name vnet-hub \ -n dns-resolver-in --address-prefix 10.0.1.0/28
-g / --resource-groupResource group of the parent VNet. --vnet-nameHub VNet — vnet-hub. -n / --nameUser-defined name — dns-resolver-in reflects the inbound direction. --address-prefix/28 gives 11 usable IPs — far more than an inbound endpoint consumes (1–2 IPs).
9

Create the DNS Resolver outbound subnet · forwards Azure queries to on-prem DNS

The outbound endpoint forwards queries from Azure resources to your on-premises DNS server. This lets Databricks workers and Azure VMs resolve private on-prem hostnames — such as your SQL Server — without leaving the VNet.

Purpose
Forwards Azure VNet DNS queries to on-prem DNS servers via a forwarding ruleset — resolves internal hostnames not visible in Azure Private DNS zones.
Delegation
Delegated to Microsoft.Network/dnsResolvers when the outbound endpoint is provisioned — same mechanism as the inbound subnet.
az network vnet subnet create -g lakeflow-mssql-on-prem --vnet-name vnet-hub \ -n dns-resolver-out --address-prefix 10.0.1.16/28
-g / --resource-groupResource group of the parent VNet. --vnet-nameHub VNet — vnet-hub. -n / --nameUser-defined name — dns-resolver-out for the outbound direction. --address-prefix/28. Choose a non-overlapping block within vnet-hub's address space.
10

Create the Azure DNS Private Resolver + inbound endpoint · forwards on-prem privatelink queries into Azure

The inbound endpoint is the IP address your on-premises DNS server forwards Azure private DNS queries to. Queries arrive over the VPN tunnel, the resolver passes them to Azure-provided DNS (168.63.129.16), which matches Private DNS zones linked to vnet-hub and returns Private Endpoint IPs.

Inbound endpoint IP
Azure assigns a dynamic private IP from dns-resolver-in (step 8). Step 10d captures it — this becomes the forward-to address in step 11.
Why delegated subnet?
The inbound endpoint NIC must live in a subnet delegated to Microsoft.Network/dnsResolvers — that delegation was applied when the endpoint is provisioned, using the subnet created in step 8.
# 10a. Create the resolver shell attached to vnet-hub az dns-resolver create -g lakeflow-mssql-on-prem -n dns-resolver \ --id $(az network vnet show -g lakeflow-mssql-on-prem -n vnet-hub --query id -o tsv) # 10b. Capture the delegated subnet resource ID SUBNET_ID=$(az network vnet subnet show -g lakeflow-mssql-on-prem --vnet-name vnet-hub \ -n dns-resolver-in --query id -o tsv) # 10c. Create the inbound endpoint az dns-resolver inbound-endpoint create \ -g lakeflow-mssql-on-prem --dns-resolver-name dns-resolver \ -n in-endpoint \ --ip-configurations "[{private-ip-allocation-method:'Dynamic',id:'$SUBNET_ID'}]" # 10d. Capture assigned IP — enter this in step 11 az dns-resolver inbound-endpoint show \ -g lakeflow-mssql-on-prem --dns-resolver-name dns-resolver \ -n in-endpoint --query "ipConfigurations[0].privateIpAddress" -o tsv
⚠ CLI quirk — AAZ-generated az dns-resolver schema
Pass the subnet resource ID as id: at the top level of each --ip-configurations element — not nested inside subnet:{id:...}. The nested form appears in older docs and now fails with "Model 'AAZObjectArg' has no field named 'subnet'". Run az dns-resolver inbound-endpoint create --help to confirm your CLI version's schema.
az dns-resolver create --idFull ARM resource ID of the VNet the resolver is scoped to. The resolver shell has no IP itself — endpoints do. --ip-configurationsJSON array with id (subnet resource ID) and private-ip-allocation-method. Dynamic = Azure picks the IP from the subnet range. inbound-endpoint show --queryExtracts the assigned private IP. Note this — it becomes the forward-to address configured on your on-prem DNS in step 11.
11

Configure on-prem DNS conditional forwarder · UDM Pro or AdGuard Home · verify end-to-end

Point your on-premises DNS server at the inbound endpoint IP (from step 10d) for all Azure Private DNS suffixes. Queries for *.azuredatabricks.net, *.blob.core.windows.net, and similar will now resolve to Private Endpoint IPs instead of public ones.

UDM Pro
UniFi Network → Settings → Routing → DNS → Conditional forwarder. Also add the hub VNet CIDR (e.g. 10.0.0.0/26) to the VPN site's Remote Networks so the UDM can reach the inbound endpoint IP.
AdGuard Home
Settings → DNS → Upstream DNS servers. Paste domain-specific upstreams using the [/domain/]IP format at the top of the list, above any fallback resolvers.
# --- Option A: UDM Pro --- # UniFi Network → Settings → Routing → DNS → Conditional forwarder privatelink.azuredatabricks.net → <inbound-endpoint-IP> azuredatabricks.net → <inbound-endpoint-IP> # --- Option B: AdGuard Home --- # Settings → DNS → Upstream DNS servers (paste at top): [/azuredatabricks.net/privatelink.azuredatabricks.net/]<inbound-endpoint-IP> [/privatelink.blob.core.windows.net/privatelink.dfs.core.windows.net/]<inbound-endpoint-IP> [/privatelink.vaultcore.azure.net/privatelink.database.windows.net/]<inbound-endpoint-IP> # Verify — flush client DNS cache first, then confirm private resolution ipconfig /flushdns nslookup adb-NNN...azuredatabricks.net
⚠ Add hub VNet CIDR to UDM Remote Networks
The inbound endpoint IP is in vnet-hub's address space. If that CIDR isn't listed in the VPN site's Remote Networks, the UDM and AdGuard cannot route DNS queries to it over the IPsec tunnel — they'll time out silently.
Conditional forwarderSends queries for specific domains to a designated DNS server instead of the default recursive resolver. Only Azure private suffixes route to the inbound endpoint — all other DNS remains unchanged. [/domain/]IP syntaxAdGuard Home per-domain upstream format. Each [/…/] block lists domain suffixes that route to the IP after the closing bracket. Multiple suffixes separated by /. nslookup verifyShould return the Private Endpoint's private IP (e.g. 10.139.0.4), not the public Azure IP. If still public, flush DNS, wait 30 s, and retry.
12

Create the Workspace PE subnet in the Databricks spoke · Databricks UI/API & browser-auth endpoints

Dedicated subnet in vnet-dbx for Databricks UI/API and browser-auth Private Endpoints. Separating these from back-end PEs keeps NSG rules and diagnostic filtering cleaner.

PE targets
Databricks workspace UI (*.azuredatabricks.net) and browser auth PE — user-facing endpoints only.
Why separate from pe-subnet?
Isolating user-facing PEs from data-plane PEs (ADLS, KV, SQL) simplifies NSG scoping and diagnostic filtering.
az network vnet subnet create -g lakeflow-mssql-on-prem --vnet-name vnet-dbx -n workspace-pe-subnet \ --address-prefix 10.0.0.112/28 \ --private-endpoint-network-policies Disabled
--vnet-name vnet-dbxDatabricks spoke VNet — created in the VNet & Subnet guide step 1. --address-prefix10.0.0.112/28 — last /28 in the spoke's 10.0.0.64/26 address space. 11 usable IPs; each PE consumes one. --private-endpoint-network-policies DisabledMandatory — Azure blocks PE NIC injection in subnets where NSG or UDR network policies are still active.
13

Capture hub and spoke VNet resource IDs · required for VNet peering in the VNet guide

VNet peering requires the full ARM resource ID of the remote VNet — the short name is not accepted cross-subscription and is fragile even within the same resource group. Capture both into shell variables before running the peering commands.

Why full ID?
--remote-vnet works with the short name only when both VNets share a resource group in the same subscription. The full ARM ID is always safe.
CMD equivalent
for /f %i in ('az network vnet show ... --query id -o tsv') do set HUB=%i — see warn below.
HUB=$(az network vnet show -g lakeflow-mssql-on-prem -n vnet-hub --query id -o tsv) DBX=$(az network vnet show -g lakeflow-mssql-on-prem -n vnet-dbx --query id -o tsv)
⚠ Windows CMD — different syntax
for /f %i in ('az network vnet show -g lakeflow-mssql-on-prem -n vnet-hub --query id -o tsv') do set HUB=%i
for /f %i in ('az network vnet show -g lakeflow-mssql-on-prem -n vnet-dbx --query id -o tsv') do set DBX=%i
--query idJMESPath expression — extracts the id field (full ARM resource ID) from the JSON response. -o tsvStrips JSON quoting so the raw string is captured cleanly by $() subshell. Use with for /f on Windows CMD.
14

Peer hub ↔ spoke VNets · gateway transit + remote gateway routing

Two peering commands — one from each side. The hub side advertises its VPN Gateway (--allow-gateway-transit); the spoke side uses it (--use-remote-gateways). Step 13's $HUB and $DBX variables must be set in the same shell session before running these.

--allow-gateway-transit
Set on the hub side. Advertises the hub's VPN Gateway to the spoke — spoke traffic destined for on-prem routes through it automatically.
--use-remote-gateways
Set on the spoke side. Fails if the hub peering is not yet in Connected state with allowGatewayTransit: true — run hub side first.
# Hub → Spoke (must run first) az network vnet peering create -g lakeflow-mssql-on-prem --vnet-name vnet-hub -n hub-to-dbx \ --remote-vnet $DBX \ --allow-vnet-access \ --allow-forwarded-traffic \ --allow-gateway-transit # Spoke → Hub (run after hub peering reaches Connected) az network vnet peering create -g lakeflow-mssql-on-prem --vnet-name vnet-dbx -n dbx-to-hub \ --remote-vnet $HUB \ --allow-vnet-access \ --allow-forwarded-traffic \ --use-remote-gateways
--allow-vnet-accessPermits VM-to-VM and service traffic between the peered VNets. Required on both sides for any connectivity. --allow-forwarded-trafficAccepts packets that didn't originate in this VNet — needed when the hub routes spoke egress through a gateway or NVA. --allow-gateway-transitHub-side only. Exposes the hub VPN Gateway to the spoke. Must be set before the spoke-side --use-remote-gateways command runs. --use-remote-gatewaysSpoke-side only. Routes spoke traffic for on-prem CIDRs through the hub VPN Gateway. Cannot coexist with --allow-gateway-transit on the same peering.
Configuring AdGuard DNS Settings

AdGuard Home — Upstream DNS per Domain

Settings → DNS → Upstream DNS servers. Paste the block below at the top of the upstream list, above any fallback resolvers. Replace <inbound-endpoint-ip> with the private IP captured in step 10d. AdGuard evaluates rules top-to-bottom — the [/domain/]IP entries take priority over the default resolver for matched suffixes only; all other DNS remains unchanged.

# Databricks workspace UI, API, browser auth [/azuredatabricks.net/]<inbound-endpoint-ip> # ADLS Gen2 blob, DFS, Files, Queue, Table [/blob.core.windows.net/dfs.core.windows.net/file.core.windows.net/queue.core.windows.net/table.core.windows.net/]<inbound-endpoint-ip> # Key Vault [/vaultcore.azure.net/vault.azure.net/]<inbound-endpoint-ip> # Azure SQL Database / Managed Instance [/database.windows.net/]<inbound-endpoint-ip> # Event Hubs / Service Bus [/servicebus.windows.net/]<inbound-endpoint-ip> # Azure Container Registry [/azurecr.io/]<inbound-endpoint-ip>
Format
[/suffix.../]IP — AdGuard per-domain upstream. Multiple suffixes in one rule separated by /. Trailing /] closes the match list.
Inbound endpoint IP
Captured in step 10d — a dynamic private IP from the dns-resolver-in subnet (10.0.1.0/28). Not 172.16.4.4 — that was the old network.
privatelink.* covered?
Yes — suffix matching catches *.privatelink.azuredatabricks.net, *.privatelink.blob.core.windows.net, etc. without needing explicit entries.
Placement
Must be above any wildcard or fallback upstream (e.g. 9.9.9.9). AdGuard stops at the first matching rule.
VPN tunnel prerequisite
The inbound endpoint IP is only reachable if the S2S tunnel is up and vnet-hub's CIDR is listed in UDM Remote Networks.
⚠ Update the IP after any DNS Resolver re-deployment. The inbound endpoint IP is dynamically assigned — if you delete and recreate the resolver or endpoint, the IP changes. Re-run the az dns-resolver inbound-endpoint show … --query ipConfigurations[0].privateIpAddress command from step 10d and update every occurrence in AdGuard.

Spin Up a Test VM & Verify End-to-End Connectivity
A

Create an NSG scoped to your on-prem subnet

Create a dedicated NSG that allows ICMP and SSH only from your on-prem LAN. No public inbound access — the VM will be reachable exclusively through the VPN tunnel.

# Create the NSG az network nsg create -g lakeflow-mssql-on-prem -n nsg-test-vm # Allow ICMP (ping) from on-prem only az network nsg rule create -g lakeflow-mssql-on-prem --nsg-name nsg-test-vm -n allow-icmp-onprem \ --priority 100 --protocol Icmp --direction Inbound \ --source-address-prefixes 192.168.1.0/24 --destination-port-ranges "*" # Allow SSH (port 22) from on-prem only az network nsg rule create -g lakeflow-mssql-on-prem --nsg-name nsg-test-vm -n allow-ssh-onprem \ --priority 110 --protocol Tcp --direction Inbound \ --source-address-prefixes 192.168.1.0/24 --destination-port-ranges 22
--source-address-prefixesLocks inbound access to your on-prem LAN only. Replace 192.168.1.0/24 if your LAN subnet differs. --priorityLower number = higher priority. Keep ICMP at 100 and SSH at 110 so both are evaluated before the default deny-all rule at 65500.

Basic SKU (PolicyBased): uses a crypto map — no virtual tunnel interface (VTI) is created, so the Tunnel IP field is left disabled and 169.254.21.2 does not apply.

VpnGw1+ (RouteBased): Azure reserves APIPA addresses 169.254.21.1 / 169.254.22.1 as its BGP peer IPs and expects your device to use 169.254.21.2 / 169.254.22.2 as the local VTI endpoint. Enable Tunnel IP, set IPv4 to 169.254.21.2, Netmask /30. These are link-local only — not routed across the tunnel. Remote Networks still holds your Azure VNet CIDR.

⚠ Gotchas Specific to UDM Pro

  • Dynamic WAN IP: consumer ISPs rotate your public IP. Azure's LocalNetworkGateway needs static. Options: get a static IP from your ISP, or move to VpnGw1+ and use FQDN-based LNG with DDNS (not supported on Basic).
  • ISP modem in NAT mode: put it in bridge mode so the UDM holds the public IP. Otherwise double-NAT breaks NAT-T. If bridging isn't possible, port-forward UDP 500 & 4500 + ESP to the UDM.
  • Weak crypto on Basic SKU: SHA1 + DH Group 2 fails most compliance frameworks. For anything beyond a homelab, upgrade the Azure gateway to VpnGw1AZ and switch UniFi to AES-256-GCM / SHA-256 / DH 14.
  • UniFi OS version: older firmware (< UniFi OS 3.x / Network 7.x) has limited Manual IPsec options. Upgrade controller and UDM firmware first.
  • MTU: if throughput is poor or some apps hang, set MSS clamping to 1350 on the IPsec interface (Settings → Internet → Advanced).

🔧 Troubleshooting Playbook — UDM Pro ⇄ Azure

🚫 Tunnel stuck on "Connecting" — Phase 1 never completes
Likely: PSK mismatch (watch UDM's 22-char limit), clock skew, or ISP blocking UDP 500/4500. Check: SSH to UDM & run ipsec statusall. Azure side: enable IKEDiagnosticLog, grep for "PSK" / "no matching proposal".
⚙️ Phase 1 up, Phase 2 fails
Likely: crypto or selector mismatch. Fix: confirm both sides are Route Based (or both Policy Based); align Phase 2 transform (AES-256 / SHA1 for Basic, AES-256 / SHA-256 for VpnGw*). Lifetime must match — 3600 s.
🚷 Tunnel green but can't reach Azure VMs
Check: UDM firewall LAN↔IPsec rule exists; ip route shows Azure CIDR via VTI; Azure NSG allows inbound from your LAN CIDR; VM's OS firewall (Windows Defender) not blocking ICMP/RDP/SSH.
⏱️ Tunnel drops every ~60 min
Likely: Phase 2 rekey failing. Fix: align Phase 2 lifetime to 3600 s on both sides, enable DPD (delay 10 s, action Restart). Disable "replay window" on strict devices. ISP idle-timeout < rekey also causes this — enable keepalive.
🌀 Tunnel drops whenever ISP rotates WAN IP
Fix: Azure LNG is pinned to the old IP. Update it: az network local-gateway update -g lakeflow-mssql-on-prem -n lng-onprem --gateway-ip-address <new>. Permanent fix: static IP from ISP, or move to VpnGw1+ and use FQDN-based LNG with DDNS.
🐢 Slow throughput / random app hangs over tunnel
Likely: MTU/MSS fragmentation. Fix: set MSS clamp to 1350 on UDM (Settings → Internet → Advanced). On Azure side ensure VM NIC accelerated networking is on. Also check double-NAT — ISP modem should be bridged.
