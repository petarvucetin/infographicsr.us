Before You Start — Prerequisites Checklist
📋

Non-Overlapping CIDRs

Your on-prem space (e.g. 10.0.0.0/16) and Azure VNet (e.g. 172.16.0.0/16) must not overlap — not even partially.

🌐

Static Public IP on-prem

VPN device needs a routable, static public IPv4. Dynamic IP works only with FQDN on some SKUs — avoid for production.

🔥

Firewall Ports Open

Allow outbound UDP 500 & UDP 4500 and IP protocol 50 (ESP). Same on any upstream ISP/edge firewall.

🛠️

Compatible VPN Device

Check Microsoft's validated-devices list. Prefer IKEv2, route-based. Have admin access & latest firmware.

The 7-Step Build — Azure Side, Then On-Prem
1

Create (or identify) the Azure Virtual Network · Azure Portal or CLI

Your hub VNet that will host the gateway. Use an address space that does not overlap with anything on-prem or in other peered VNets.

Resource
Microsoft.Network/virtualNetworks
Example address space
172.16.0.0/16 with workload subnets like 172.16.1.0/24
# Azure CLI az network vnet create -g rg-hub -n vnet-hub --address-prefix 172.16.0.0/16 --subnet-name snet-app --subnet-prefix 172.16.1.0/24
2

Add the GatewaySubnet · /27 minimum, /26 recommended

A dedicated subnet named exactly GatewaySubnet. Do not attach an NSG. Do not add a UDR that overrides 0.0.0.0/0.

Why /26?
Leaves room for future Route Server, ExpressRoute gateway coexistence, or larger SKUs that need more IPs.
Example prefix
172.16.255.0/26 (high end of VNet)
az network vnet subnet create -g rg-hub --vnet-name vnet-hub -n GatewaySubnet --address-prefix 172.16.255.0/26
3

Create a Public IP for the VPN Gateway · Standard SKU · Static

Standard-SKU Static public IP is required for all modern VpnGw* SKUs. For active-active, create two public IPs and attach both.

az network public-ip create -g rg-hub -n pip-vpngw-01 --sku Standard --allocation-method Static --zone 1 2 3
4

Create the VPN Gateway · type=Vpn, vpnType=RouteBased — takes 20–45 min

Pick a VpnGw1 AZ or larger for any production workload. This step is the slow one — Azure provisions redundant VMs behind the scenes.

Recommended SKU
VpnGw2AZ route-based — HA, BGP-capable, zone-redundant
Active-active?
Pass --active-active true and two public IPs to get dual instances with BGP
az network vnet-gateway create -g rg-hub -n vpngw-hub \ --vnet vnet-hub --public-ip-addresses pip-vpngw-01 \ --gateway-type Vpn --vpn-type RouteBased --sku VpnGw2AZ --no-wait
5

Create the Local Network Gateway · Azure's representation of your on-prem

This resource is just metadata: on-prem public IP + the CIDRs reachable behind it (or BGP peer IP/ASN if you're using BGP).

az network local-gateway create -g rg-hub -n lng-onprem \ --gateway-ip-address 198.51.100.10 \ --local-address-prefixes 10.0.0.0/16 10.1.0.0/16
6

Create the Connection + Shared Key · Ties gateway + LNG together

Generate a strong PSK (32+ random chars). Store it in Key Vault. Use the same key exactly on the on-prem device. Optionally set an IPsec/IKE policy to force modern crypto.

# Generate a PSK openssl rand -base64 32 # Create connection az network vpn-connection create -g rg-hub -n cn-onprem \ --vnet-gateway1 vpngw-hub --local-gateway2 lng-onprem \ --shared-key "<PSK>" --enable-bgp false
7

Configure the On-Prem VPN Device · Download Azure's config template

From the Connection blade in the portal, click Download configuration. Pick your device vendor/model/firmware — Azure spits out a vendor-specific config with all the right IKE/IPsec params. Import, adjust interface names, commit.

Key bits to match
Peer IP = Azure gateway public IP · PSK · IKEv2 · AES-256 + SHA-256 · DH 14 · Lifetime 28800 / 3600
Route traffic
Add static route on-prem: 172.16.0.0/16 → tunnel interface (or let BGP do it)
Validate — Prove the Tunnel Is Really Up
✅

Connection Status

Azure Portal → Connection → should show Connected with non-zero data in/out.

az network vpn-connection show -g rg-hub -n cn-onprem --query connectionStatus

📡

End-to-End Ping

From on-prem host → VM in Azure VNet (NSG must allow ICMP). From Azure VM → on-prem host.

ping 172.16.1.4 · Test-NetConnection 10.0.5.20 -Port 3389

📈

Monitoring & Alerts

Enable diagnostic logs (GatewayDiagnosticLog, IKEDiagnosticLog, TunnelDiagnosticLog). Create alerts on TunnelConnectionStatus metric & TunnelEgressPacketDropCount.
