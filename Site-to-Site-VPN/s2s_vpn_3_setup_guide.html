Before You Start — Prerequisites Checklist
📋

Non-Overlapping CIDRs

Your on-prem space (e.g. 192.168.0.0/24) and Azure VNet (e.g. 10.0.0.0/26) must not overlap — not even partially.

🌐

Static Public IP on-prem

VPN device needs a routable, static public IPv4. Dynamic IP works only with FQDN on some SKUs — avoid for production.

🔥

Firewall Ports Open

Allow outbound UDP 500 & UDP 4500 and IP protocol 50 (ESP). Same on any upstream ISP/edge firewall.

🛠️

Compatible VPN Device

Check Microsoft's validated-devices list. Prefer IKEv2, route-based. Have admin access & latest firmware.

The 7-Step Build — Azure Side, Then On-Prem
1

Create (or identify) the Azure Virtual Network · Azure Portal or CLI

Your hub VNet that will host the gateway. Use an address space that does not overlap with anything on-prem or in other peered VNets.

Resource
Microsoft.Network/virtualNetworks
Example address space
10.0.0.0/26 with workload subnets like 10.0.0.0/27
# Azure CLI az network vnet create -g rg-hub -n vnet-hub --address-prefix 10.0.0.0/26 --subnet-name snet-app --subnet-prefix 10.0.0.0/27
-g / --resource-groupResource group the VNet is created in. Must already exist. -n / --nameName of the virtual network resource. -l / --locationOmitted — inherited from the resource group's location automatically. --address-prefixCIDR of the entire VNet. Must not overlap on-prem or peered VNets. --subnet-nameName of a first subnet to create alongside the VNet (convenience — optional). --subnet-prefixCIDR of that first subnet — must fit inside --address-prefix.
2

Add the GatewaySubnet · /27 minimum, /26 recommended

A dedicated subnet named exactly GatewaySubnet. Do not attach an NSG. Do not add a UDR that overrides 0.0.0.0/0.

Why /26?
Leaves room for future Route Server, ExpressRoute gateway coexistence, or larger SKUs that need more IPs.
Example prefix
10.0.0.32/27 (upper half of VNet)
az network vnet subnet create -g rg-hub --vnet-name vnet-hub -n GatewaySubnet --address-prefix 10.0.0.32/27
-g / --resource-groupResource group of the parent VNet. --vnet-nameVNet to add this subnet to. -n / --nameSubnet name. For the VPN gateway subnet it must be exactly GatewaySubnet — Azure looks it up by this name. --address-prefixSubnet CIDR inside the VNet's address space. /27 min, /26 recommended.
3

Create a Public IP for the VPN Gateway · Basic SKU · Dynamic

The Basic VPN Gateway SKU requires a Basic-SKU Dynamic public IP. Standard/Static IPs are only needed for VpnGw1+ SKUs.

az network public-ip create -g rg-hub -n pip-vpngw-01 --sku Basic --allocation-method Dynamic
-g / --resource-groupResource group for the Public IP. -n / --nameName of the Public IP resource. --skuBasic — required for the Basic VPN Gateway SKU. Upgrade to Standard when moving to VpnGw1+. --allocation-methodDynamic — Basic-SKU public IPs must be dynamic. The actual IP is assigned when the gateway is provisioned.
4

Create the VPN Gateway · type=Vpn, vpnType=PolicyBased — takes 20–25 min

The Basic SKU is the lowest-cost option — suitable for dev/test and low-throughput S2S. No BGP, no active-active, no zone redundancy. This step is the slow one — Azure provisions the gateway VM behind the scenes.

SKU
Basic — ~100 Mbps, up to 10 S2S tunnels, PolicyBased or RouteBased
Active-active?
Not supported on Basic. Upgrade to VpnGw1+ for HA / active-active
az network vnet-gateway create -g rg-hub -n vpngw-hub \ --vnet vnet-hub --public-ip-addresses pip-vpngw-01 \ --gateway-type Vpn --vpn-type PolicyBased --sku Basic --no-wait
-g / --resource-groupResource group for the VPN Gateway. -n / --nameVPN Gateway resource name. --vnetVNet the gateway lives in. It attaches automatically to that VNet's GatewaySubnet. --public-ip-addressesThe Basic-SKU Dynamic public IP created in step 3. --gateway-typeVpn for Site-to-Site / P2S. Alternatives: ExpressRoute, LocalGateway. --vpn-typePolicyBased (IKEv1, static routing — Basic only) or RouteBased (IKEv2, also supported on Basic but with limited features). --skuBasic — lowest cost tier. Upgrade to VpnGw1–VpnGw5 (append AZ for zone-redundancy) for production workloads. --no-waitReturn immediately instead of polling. Provisioning runs for 20–25 min in the background.
5

Create the Local Network Gateway · Azure's representation of your on-prem

This resource is just metadata: on-prem public IP + the CIDRs reachable behind it (or BGP peer IP/ASN if you're using BGP).

⚠ Replace both placeholder values before running this command
  1. 198.51.100.10 → your on-prem VPN device's actual public WAN IP. Find it on your router's WAN status page, or run curl ifconfig.me from any on-prem host. Using the wrong IP here is the most common reason the tunnel gets stuck — Azure will silently drop all IKE packets from an unrecognised source.
  2. 192.168.0.0/24 → the specific LAN subnet(s) you want to route through the tunnel (e.g. 192.168.1.0/24). Only list subnets that need access to Azure. To update later: az network local-gateway update -g <rg> -n lng-onprem --local-address-prefixes <CIDRs>
az network local-gateway create -g rg-hub -n lng-onprem \ --gateway-ip-address 198.51.100.10 \ --local-address-prefixes 192.168.0.0/24
-g / --resource-groupResource group for the Local Network Gateway. -n / --nameLNG resource name. This is just metadata that describes your on-prem side. --gateway-ip-address⚠ Placeholder — replace 198.51.100.10 with your VPN device's real public WAN IP. Azure uses this to identify your device; a wrong value causes it to silently reject all incoming IKE packets. --local-address-prefixes⚠ Placeholder — replace 192.168.0.0/24 with your actual LAN subnet(s). Space-separated for multiple. Must match what your on-prem device advertises as its local traffic selector.
6

Create the Connection + Shared Key · Ties gateway + LNG together

Generate a strong PSK (32+ random chars). Store it in Key Vault. Use the same key exactly on the on-prem device. Optionally set an IPsec/IKE policy to force modern crypto.

# Generate a PSK openssl rand -base64 32 # Create connection az network vpn-connection create -g rg-hub -n cn-onprem \ --vnet-gateway1 vpngw-hub --local-gateway2 lng-onprem \ --shared-key "<PSK>" # Add --enable-bgp (no value) if you want dynamic routing
-g / --resource-groupResource group for the Connection resource. -n / --nameConnection resource name. One connection = one tunnel pair. --vnet-gateway1Azure-side VPN Gateway (side 1 of the tunnel). --local-gateway2On-prem Local Network Gateway (side 2). VNetGw + LNG pairing = S2S/IPsec connection type. --shared-keyPre-shared key (PSK). 32+ random chars. Must match on-prem device exactly. Store in Key Vault. --enable-bgpFlag only — include --enable-bgp (no value) to turn on dynamic BGP route exchange. Omit it for static routing (the default). Passing --enable-bgp false fails with "unrecognized arguments: false".
7

Configure the On-Prem VPN Device · Manual config — Basic SKU

Note: Download configuration (Connection blade → portal) is not available for the Basic SKU. Configure your on-prem device manually using the values below. The template download is only supported on VpnGw1 and higher.

Key bits to match
Peer IP = Azure gateway public IP · PSK · IKEv1 · AES-256 + SHA-1 · DH Group 2 · Lifetime 28800 / 3600
Route traffic
Add static route on-prem: 10.0.0.0/26 → tunnel interface (no BGP on Basic)

Spin Up a Test VM & Verify End-to-End Connectivity
A

Create an NSG scoped to your on-prem subnet

Create a dedicated NSG that allows ICMP and SSH only from your on-prem LAN. No public inbound access — the VM will be reachable exclusively through the VPN tunnel.

# Create the NSG az network nsg create -g rg-hub -n nsg-test-vm # Allow ICMP (ping) from on-prem only az network nsg rule create -g rg-hub --nsg-name nsg-test-vm -n allow-icmp-onprem \ --priority 100 --protocol Icmp --direction Inbound \ --source-address-prefixes 192.168.1.0/24 --destination-port-ranges "*" # Allow SSH (port 22) from on-prem only az network nsg rule create -g rg-hub --nsg-name nsg-test-vm -n allow-ssh-onprem \ --priority 110 --protocol Tcp --direction Inbound \ --source-address-prefixes 192.168.1.0/24 --destination-port-ranges 22
--source-address-prefixesLocks inbound access to your on-prem LAN only. Replace 192.168.1.0/24 if your LAN subnet differs. --priorityLower number = higher priority. Keep ICMP at 100 and SSH at 110 so both are evaluated before the default deny-all rule at 65500.
B

Create the test VM · Ubuntu 22.04 · No public IP · Standard_B1s

Ubuntu is used because ICMP and SSH work out of the box — no OS firewall changes needed. --public-ip-address "" ensures the VM has no internet exposure; it is reachable only through the tunnel.

⚠ Replace <StrongPassword> before running
Must be 12+ characters with uppercase, lowercase, digit, and symbol. Azure enforces complexity at deployment and will reject the command if it fails. Example: Tunnel@Test2026!
az vm create -g rg-hub -n vm-vpn-test \ --image Ubuntu2204 \ --vnet-name vnet-hub --subnet snet-app \ --nsg nsg-test-vm \ --admin-username azureuser \ --admin-password "<StrongPassword>" \ --size Standard_B1s \ --public-ip-address ""
--image Ubuntu2204Ubuntu 22.04 LTS — ICMP and SSH work immediately, no OS firewall changes needed. Use Win2022Datacenter if you need Windows, but you'll need to enable ICMP in Windows Firewall manually after deployment. --nsg nsg-test-vmAttaches the NSG from step A, restricting inbound access to the on-prem subnet only. --public-ip-address ""Empty string suppresses public IP creation. VM is only reachable via the VPN — this is intentional. --size Standard_B1sCheapest burstable SKU (1 vCPU, 1 GB RAM). More than enough for a connectivity test. Remember to delete it when done — it accrues hourly charges while running.
C

Get the VM's private IP

The IP will be assigned from the 10.0.0.0/27 workload subnet. Note it down for the tests below.

az vm show -g rg-hub -n vm-vpn-test -d --query privateIps -o tsv
D

Test from an on-prem host · Must be on 192.168.1.x

Run these from a machine on your 192.168.1.0/24 LAN. Both should succeed if the tunnel and NSG are configured correctly.

Ping (ICMP)
ping <vm-private-ip>
Expect replies in <10 ms. Timeout = tunnel not routing or NSG blocking ICMP.
SSH
ssh azureuser@<vm-private-ip>
Successful login confirms full TCP connectivity through the tunnel.
⚠ If both fail
1. Confirm the tunnel is up: az network vpn-connection show -g rg-hub -n cn-onprem --query connectionStatus
2. Confirm your host is on the subnet listed in the LNG: az network local-gateway show -g rg-hub -n lng-onprem --query localNetworkAddressSpace
3. Confirm the VM's NIC has the NSG attached: az vm show -g rg-hub -n vm-vpn-test --query networkProfile
E

Clean up — delete test resources when done

The VM accrues compute charges while running. Delete it once testing is complete.

# Delete VM (confirm immediately, run in background) az vm delete -g rg-hub -n vm-vpn-test --yes --no-wait # Delete the NSG az network nsg delete -g rg-hub -n nsg-test-vm

⚠ az vm delete removes the VM but leaves the OS disk and NIC by default. Clean those up manually in the portal under rg-hub, or run: az disk delete -g rg-hub -n vm-vpn-test_OsDisk_1_* --yes and az network nic delete -g rg-hub -n vm-vpn-testVMNic --no-wait

Worked Example — UniFi UDM Pro (Homelab / SMB)
🏠

UniFi Dream Machine Pro  ⇄  Azure VPN Gateway Basic SKU

Not on Microsoft's validated-devices list — no auto-generated config template. Configure manually via the UniFi Network app. Values below match Azure's Basic SKU default IPsec/IKE policy.

1 Navigate in UniFi Network

  1. Open the UniFi Network app → log in to your UDM Pro
  2. Go to Settings → VPN
  3. Site-to-Site VPN → Create New
  4. VPN Type: Manual IPsec
  5. Name it e.g. azure-hub

2 Top-Level Tunnel Fields

  • Enabled: ✓
  • Remote Gateway: Azure VPN Gateway public IP (from pip-vpngw-01)
  • Remote Subnets: your Azure VNet CIDR, e.g. 10.0.0.0/26
  • Local Subnets: your LAN CIDR, e.g. 192.168.0.0/24
  • Pre-Shared Key: the PSK you set on the Azure Connection
  • Local WAN IP: auto-detect (or specify if multi-WAN)
  • Route Distance: leave default (30)

Basic SKU (PolicyBased): uses a crypto map — no virtual tunnel interface (VTI) is created, so the Tunnel IP field is left disabled and 169.254.21.2 does not apply.

VpnGw1+ (RouteBased): Azure reserves APIPA addresses 169.254.21.1 / 169.254.22.1 as its BGP peer IPs and expects your device to use 169.254.21.2 / 169.254.22.2 as the local VTI endpoint. Enable Tunnel IP, set IPv4 to 169.254.21.2, Netmask /30. These are link-local only — not routed across the tunnel. Remote Networks still holds your Azure VNet CIDR.

5 Firewall & Routing on UDM

  • Firewall → allow traffic between LAN ↔ the new IPsec interface
  • Auto-route should add 10.0.0.0/26 → IPsec. Verify under Routing
  • Check tunnel status: Insights → VPN or SSH & run ipsec status

⚠ Gotchas Specific to UDM Pro

  • Dynamic WAN IP: consumer ISPs rotate your public IP. Azure's LocalNetworkGateway needs static. Options: get a static IP from your ISP, or move to VpnGw1+ and use FQDN-based LNG with DDNS (not supported on Basic).
  • ISP modem in NAT mode: put it in bridge mode so the UDM holds the public IP. Otherwise double-NAT breaks NAT-T. If bridging isn't possible, port-forward UDP 500 & 4500 + ESP to the UDM.
  • Weak crypto on Basic SKU: SHA1 + DH Group 2 fails most compliance frameworks. For anything beyond a homelab, upgrade the Azure gateway to VpnGw1AZ and switch UniFi to AES-256-GCM / SHA-256 / DH 14.
  • UniFi OS version: older firmware (< UniFi OS 3.x / Network 7.x) has limited Manual IPsec options. Upgrade controller and UDM firmware first.
  • MTU: if throughput is poor or some apps hang, set MSS clamping to 1350 on the IPsec interface (Settings → Internet → Advanced).

🔧 Troubleshooting Playbook — UDM Pro ⇄ Azure

🚫 Tunnel stuck on "Connecting" — Phase 1 never completes
Likely: PSK mismatch (watch UDM's 22-char limit), clock skew, or ISP blocking UDP 500/4500. Check: SSH to UDM & run ipsec statusall. Azure side: enable IKEDiagnosticLog, grep for "PSK" / "no matching proposal".
⚙️ Phase 1 up, Phase 2 fails
Likely: crypto or selector mismatch. Fix: confirm both sides are Route Based (or both Policy Based); align Phase 2 transform (AES-256 / SHA1 for Basic, AES-256 / SHA-256 for VpnGw*). Lifetime must match — 3600 s.
🚷 Tunnel green but can't reach Azure VMs
Check: UDM firewall LAN↔IPsec rule exists; ip route shows Azure CIDR via VTI; Azure NSG allows inbound from your LAN CIDR; VM's OS firewall (Windows Defender) not blocking ICMP/RDP/SSH.
⏱️ Tunnel drops every ~60 min
Likely: Phase 2 rekey failing. Fix: align Phase 2 lifetime to 3600 s on both sides, enable DPD (delay 10 s, action Restart). Disable "replay window" on strict devices. ISP idle-timeout < rekey also causes this — enable keepalive.
🌀 Tunnel drops whenever ISP rotates WAN IP
Fix: Azure LNG is pinned to the old IP. Update it: az network local-gateway update -g rg-hub -n lng-onprem --gateway-ip-address <new>. Permanent fix: static IP from ISP, or move to VpnGw1+ and use FQDN-based LNG with DDNS.
🐢 Slow throughput / random app hangs over tunnel
Likely: MTU/MSS fragmentation. Fix: set MSS clamp to 1350 on UDM (Settings → Internet → Advanced). On Azure side ensure VM NIC accelerated networking is on. Also check double-NAT — ISP modem should be bridged.
