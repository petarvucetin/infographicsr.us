🔵 Mode 1 — Recommended for most tables
Change Tracking (CT)
Tracks which rows changed — minimal overhead, fast, low storage impact
✅
Primary Key Required
CT only works on tables with a primary key. It records the PK of changed rows — not the full before/after column values.
⚡
Lowest Performance Impact on Source
Minimal overhead — SQL Server only tracks that a row changed, not what changed. No impact on transaction log growth.
📅
Retention Window is Critical
Set CHANGE_RETENTION high enough (recommend 2+ days) to cover any pipeline downtime window — missed changes within retention can be replayed.
ALTER DATABASE YourDB SET CHANGE_TRACKING = ON (CHANGE_RETENTION = 2 DAYS, AUTO_CLEANUP = ON); ALTER TABLE dbo.Orders ENABLE CHANGE_TRACKING WITH (TRACK_COLUMNS_UPDATED = ON);
✅ Use Change Tracking when: table has a primary key, you want minimal source DB overhead, and column-level change history is not required.
VS
🔴 Mode 2 — For tables without primary keys
Change Data Capture (CDC)
Captures every operation with full before/after column values from the transaction log
🔓
No Primary Key Required
CDC reads the SQL Server transaction log directly — captures every INSERT, UPDATE, DELETE at the column level, even for heap tables with no PK.
📋
Full Column-Level Change History
Stores before and after images of changed columns in system tables (cdc.*_CT). More storage overhead than CT, but richer change detail.
⚠️
Higher Source DB Overhead
Reads transaction log more heavily than CT. May impact source performance on very high-write tables. SQL Server Agent capture job must stay running.
EXEC sys.sp_cdc_enable_db; EXEC sys.sp_cdc_enable_table @source_schema = N'dbo', @source_name = N'Events', @role_name = NULL, @supports_net_changes = 1; -- Verify SELECT * FROM cdc.change_tables;
✅ Use CDC when: table has no primary key, you need full before/after column values, or you need complete DML history for audit/compliance purposes.

📊 Pipeline Health

  • Monitor Gateway + Ingestion Pipeline status in Databricks Jobs UI
  • Event logs in Unity Catalog: system.lakeflow.ingestion_pipeline_events
  • Set up job failure alerts via Databricks Notifications (email, PagerDuty, Slack)
  • Check pipeline run duration trends — spikes indicate source load or schema drift

⚠️ Data Freshness

  • Query MAX(_commit_timestamp) on streaming tables — should be within 2 minutes of now
  • Monitor CT retention window: if retention expires before pipeline catches up, a full re-snapshot is triggered
  • Alert if last_updated is > 5 min old via a Databricks SQL alert on system tables
  • Gateway event log shows real-time LSN / CT version progress

✅ Data Quality

  • Row count parity: compare COUNT(*) on destination vs source after each run
  • Add DLT Expectations on downstream Silver tables to catch nulls, range violations
  • Event log quality_check events track expectation failures per table per run
  • Quarantine table captures rows that fail quality expectations without blocking pipeline

💰 Cost Monitoring

  • Track Lakeflow Connect DBU usage via system.billing.usage — filter by usage_type = 'LAKEFLOW_CONNECT'
  • Gateway continuous job consumes DBUs 24/7 — optimize table selection (don't replicate unused tables)
  • Ingestion Pipeline DBU cost scales with schedule frequency and data volume
  • Set Azure / AWS budget alerts at 80% of monthly DBU budget target

🔴 Failure Recovery — Fully Automatic

  • Lakeflow Connect automatically retries failed runs with exponential backoff — no manual intervention for transient errors
  • Gateway stores last CT version / CDC LSN position — resumes exactly where it left off after a failure, preventing data gaps
  • If credentials expire, the pipeline pauses gracefully and can resume once credentials are updated in the UC Connection object
  • If CT retention window expires during an outage, Lakeflow Connect automatically triggers a full re-snapshot of affected tables
  • Re-snapshot is transparent — destination tables remain queryable during the process; they are updated atomically when complete

🔵 Production Best Practices

  • Always replicate from the primary SQL Server node — read replicas and Always On secondaries are not supported
  • Set CT CHANGE_RETENTION to at least 2x the expected maximum pipeline downtime (e.g. 2 days for weekend maintenance)
  • Deploy Gateway + Pipeline using DABs and store in Git — enables repeatable, auditable infrastructure-as-code
  • Use a dedicated SQL Server service account with db_datareader + CDC read access only — no elevated permissions
  • Schedule the Ingestion Pipeline frequency based on required data freshness — every 1 min for near-real-time, every 15 min for analytics
  • Run OPTIMIZE on destination streaming tables weekly to keep read performance fast for downstream BI and infographic tools
