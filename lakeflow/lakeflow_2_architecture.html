The Two-Stage Pipeline Design
⚡
Stage 1 · Runs Continuously
Ingestion Gateway
Extracts snapshot + CDC events from SQL Server
1
Full Snapshot on First Run
Gateway reads all rows from every selected SQL Server table and writes them to the Unity Catalog staging volume as Parquet files.
2
Continuous Change Capture
After snapshot, Gateway continuously monitors the SQL Server change log (Change Tracking or CDC) and writes new events to staging — respecting log retention windows.
3
Runs as a Continuous Serverless Job
Gateway runs 24/7 on serverless compute inside Databricks — no cluster to manage, no SHIR agent to install. Databricks auto-scales as needed.
4
Must Connect to Primary Instance
Change Tracking and CDC are not supported on read replicas or Always On secondary instances. Gateway must target the primary SQL Server node.
⬇
UC Staging Volume
⬇
🔄
Stage 2 · Runs on Schedule
Ingestion Pipeline
Applies staged data to destination streaming tables
1
Reads from Staging Volume
Pipeline reads snapshot + CDC files from the Unity Catalog staging volume written by the Gateway — decoupled, no direct SQL Server access needed here.
2
Applies Changes to Streaming Tables
Merges INSERT, UPDATE, DELETE operations into destination Unity Catalog streaming tables. Supports SCD Type 2 history tracking for slowly-changing dimensions.
3
Runs on a Custom Schedule (or Trigger)
You control the schedule — every minute, hourly, or on-demand. Lakeflow Connect auto-creates a Databricks Job for each schedule. Additional tasks can be chained.
4
Auto Schema Evolution
New source columns are automatically detected and added to destination tables. Enable auto-refresh to handle schema changes without manual pipeline updates.
🗂️
Unity Catalog Staging Volume — The Decoupling Layer
Intermediate buffer between Gateway and Ingestion Pipeline — auto-managed, governed, and version-controlled
What it stores
Full snapshot Parquet files + incremental CDC change events. Organized by table and timestamp. Retains data long enough to handle pipeline delays or replays.
What you configure
Staging catalog and schema only — Databricks automatically creates the UC volume and manages file layout. Cannot be a foreign catalog (must be a native UC catalog).
Why it matters
Decouples extraction from transformation. If the Ingestion Pipeline is delayed, the Gateway keeps capturing changes. Prevents data loss during pipeline maintenance or restarts.
On-Premises Network Connectivity Options
Azure

🔵 Azure ExpressRoute

Dedicated private circuit from on-prem network to Azure. Recommended for production. Provides consistent low latency and high bandwidth without internet exposure.

AWS

🟠 AWS Direct Connect

Dedicated private connection from on-prem to AWS. Equivalent of ExpressRoute for AWS-hosted Databricks workspaces. Bypasses public internet entirely.

Both

🟢 Site-to-Site VPN

Encrypted VPN tunnel over internet. Supported if sufficient bandwidth is available. Suitable for lower-volume workloads or dev/test environments where dedicated circuits aren't justified.

Orchestration & Job Structure

⚡ Gateway Job — Continuous Task

  • Runs 24/7 as a continuous Databricks Job task — always listening for SQL Server changes
  • Separate job from the Ingestion Pipeline — independent failure isolation
  • Auto-retries with exponential backoff on transient failures
  • Monitor progress via event logs in Unity Catalog (system.lakeflow.*)

🔄 Ingestion Pipeline — Scheduled Task

  • Runs on your chosen schedule — each schedule creates a separate Databricks Job
  • Can chain additional tasks (e.g. DLT Silver/Gold transforms) in the same job
  • Deploy via DABs (Declarative Automation Bundles) for CI/CD and source control
  • Supports SCD Type 2 history tracking on destination streaming tables
