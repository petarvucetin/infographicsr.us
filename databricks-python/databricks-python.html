01

How Python actually executes on a cluster

Your code runs in a Python REPL on the driver node — but PySpark calls don't execute there. They cross the Py4J bridge into the driver JVM, where Catalyst plans the query and the scheduler farms tasks out to executor JVMs. Python only reappears on the workers when a UDF forces it to.

Python side

The REPL is just a client. Variables, pandas frames, pyodbc cursors, and anything you collect() live in one Python process on the driver. Heavy collects are the #1 driver OOM cause.

JVM side

DataFrame ops compile to JVM/Photon. Filters, joins, aggregations, and JDBC scans never touch a Python interpreter — which is why "stay in the DataFrame API" is the core performance rule.

Crossing cost

Every UDF pays a serialization toll. Plain UDFs pickle row-by-row; pandas UDFs ship Arrow batches and are dramatically faster. If logic can't be native, make it a pandas UDF.

02

Three execution modes — and what breaks in each

The architecture in Panel 01 is the classic picture. Unity Catalog access modes and serverless compute change how much of the machinery your Python can touch. Code that pokes at JVM internals works in one mode and silently breaks in another — know which one you're targeting before you write framework code.

Dedicated (single-user)

Full classic access

Python REPL + Py4J + driver JVM exactly as drawn. spark._jvm, sparkContext, RDDs, custom JARs, init scripts — everything works.

Use for: migration jobs, anything needing JDBC tuning knobs, third-party JVM libraries, or low-level Spark configs.

Standard / shared (UC)

Sandboxed Python

Each user's Python runs in an isolated, lower-privilege sandbox so users can't read each other's data or credentials. No spark._jvm, restricted RDD API, limits on some init scripts and libraries.

Use for: multi-user analytics clusters with fine-grained UC governance.

Serverless / Spark Connect

Thin gRPC client

DBR 13+ and all serverless compute use Spark Connect: your Python is a decoupled client speaking gRPC to a remote Spark server. There is no Py4J at all — sparkContext, setLocalProperty, and JVM access don't exist.

Use for: fast-start jobs and SQL-heavy pipelines; avoid for code relying on classic internals.

iRule of thumb: write against the public DataFrame API only, and your code is portable across all three modes. Every dependency on spark._jvm, RDDs, or thread-local scheduler properties pins you to dedicated classic compute.
03

Python threading: where the GIL matters and where it doesn't

The GIL means driver threads never give you CPU parallelism in pure Python. But that's rarely what you need on Databricks — the two legitimate jobs for driver threads are submitting concurrent Spark jobs and overlapping I/O waits. Real compute parallelism belongs on the executors.

Pattern A — concurrent Spark jobs

The Spark scheduler is thread-safe. A ThreadPoolExecutor submitting actions from multiple threads runs multiple jobs simultaneously on the cluster — the canonical way to copy 20 SQL Server tables at once. Threads spend their life blocked on the JVM, so the GIL is released and irrelevant.

Pattern B — I/O fan-out

REST calls, blob ops, JDBC metadata and watermark queries, parallel dbutils.notebook.run() orchestration. All I/O-bound, all release the GIL, all thread-friendly.

Fair scheduling

Per-thread, set sc.setLocalProperty("spark.scheduler.pool", "small") so a giant fact-table copy doesn't starve dimension loads. (Classic compute only — Spark Connect has no local properties.)

Anti-pattern — multiprocessing on the driver

Forking a process that holds a live Py4J gateway is unreliable; child processes inherit broken JVM sockets. Shared and serverless compute restrict or block it outright. Don't.

Anti-pattern — CPU work in driver threads

Pure-Python number crunching across threads serializes on the GIL and saturates one driver core while the cluster idles. Push it into applyInPandas, pandas UDFs, or mapPartitions so it runs as parallel Python processes on every executor.

Shared-session gotcha

One SparkSession is shared by all threads. Session-level spark.conf.set() from thread A changes behavior for thread B. Pass per-task options through reader/writer .option() calls, never session config.

# The one threading pattern you actually need: parallel table ingestion
from concurrent.futures import ThreadPoolExecutor, as_completed

def copy_table(t):
    spark.sparkContext.setLocalProperty("spark.scheduler.pool", t["pool"])
    ingest_table(spark, jdbc_url, props, t["source"], t["target"], t.get("part_col"))
    return t["source"]

with ThreadPoolExecutor(max_workers=6) as ex:          # 4–8: protect the source DB
    futures = [ex.submit(copy_table, t) for t in manifest]
    for f in as_completed(futures):
        print("done:", f.result())                        # .result() re-raises failures
!Sizing the pool: each worker thread becomes a concurrent Spark job, and each partitioned JDBC job opens numPartitions connections to SQL Server. 6 threads × 16 partitions = 96 simultaneous source connections. Budget against the source, not the cluster.

Reaching external data: pick the right tier

There are three distinct ways Python on Databricks touches external systems, and they have completely different performance envelopes. The deciding question is always: does the data flow through the executors (parallel) or through the driver (single-threaded)?

TierWhat it isData pathUse when
Spark connectors Built-in JDBC source (SQL Server driver ships in DBR), Kafka, Event Hubs (Kafka protocol), cloud storage via abfss:// / s3:// with UC external locations. Executors, parallel. JDBC parallelizes only with explicit partition options (Panel 05). Bulk movement — millions of rows, full table copies, streaming.
Lakehouse Federation Register SQL Server / Postgres / Snowflake as UC foreign catalogs; query them like local tables with pushdown, no JDBC plumbing. Executors, with predicate pushdown to the source. Exploration, source-vs-target validation during migration, occasional joins to live source data.
Python drivers pyodbc (needs unixODBC + msodbcsql18 via init script), pymssql (no ODBC dependency), sqlalchemy, requests, azure-storage-blob, boto3. Driver only, single connection. Results land as Python objects in driver RAM. Control-plane work: watermarks, row counts, metadata, executing stored procs, small lookups.

05

Anatomy of a fast JDBC read from SQL Server

Without partition options, spark.read.jdbc opens one connection on one executor task — a 200M-row fact table crawls through a single cursor. Four options turn it into N parallel range scans.

df = (spark.read.format("jdbc")
  .option("url", jdbc_url)
  .option("dbtable", "dbo.FactSales")
  .option("user", user)
  .option("password", pwd)
  .option("driver",
    "com.microsoft.sqlserver.jdbc.SQLServerDriver")
  # ---- the four that matter ----
  .option("partitionColumn", "SalesKey")
  .option("lowerBound",  1)
  .option("upperBound",  200_000_000)
  .option("numPartitions", 16)
  # ---- tuning ----
  .option("fetchsize", 10000)
  .load())
What Spark generates

16 concurrent queries, each WHERE SalesKey >= lo AND SalesKey < hi over an even slice of the bound range — running on 16 executor tasks at once.

Choose the column carefully

partitionColumn must be numeric, date, or timestamp — ideally an indexed, evenly distributed key. A skewed column (e.g., 90% of rows in one year) gives you one giant straggler partition. Bounds don't filter rows; they only shape the split.

Fetch bounds dynamically

Run SELECT MIN(k), MAX(k) via the same JDBC source (query option) first, then feed the result in — never hard-code bounds in a reusable framework.

✓No-key tables: for heaps without a usable numeric key, use the query option with %% bucketing on a hash, or fall back to a single-threaded read for small tables — parallelism only pays for itself above a few million rows.
06

Running Python on Databricks without notebooks

A plain .py file or a packaged wheel executes as a Lakeflow Job task on a job cluster. The ingestion logic is identical to notebook code — only three plumbing details change. Everything lives in git; nothing is authored in the workspace UI.

1 · Get the session

Nothing is injected. Call SparkSession.builder.getOrCreate() — on a job cluster or serverless it attaches to the cluster's Spark.

2 · Get dbutils

No magic dbutils global. Classic compute: from pyspark.dbutils import DBUtils; DBUtils(spark). Portable (works over Spark Connect too): WorkspaceClient().dbutils from the databricks-sdk.

3 · Get parameters

No widgets. spark_python_task parameters arrive as real sys.argv — use argparse like any CLI tool. Wheel tasks receive them in the entry-point function.

# src/ingest_sqlserver.py — runs as a spark_python_task, no notebook anywhere
import argparse
from pyspark.sql import SparkSession

def main():
    p = argparse.ArgumentParser()
    p.add_argument("--table", required=True)
    p.add_argument("--target", required=True)
    p.add_argument("--partition-col", default=None)
    args = p.parse_args()

    spark = SparkSession.builder.appName("mssql-ingest").getOrCreate()
    from pyspark.dbutils import DBUtils
    pwd = DBUtils(spark).secrets.get("sqlserver", "password")   # never hard-code

    jdbc_url = ("jdbc:sqlserver://myserver.database.windows.net:1433;"
                "database=Sales;encrypt=true")
    props = {"user": "ingest_svc", "password": pwd,
             "driver": "com.microsoft.sqlserver.jdbc.SQLServerDriver"}

    ingest_table(spark, jdbc_url, props,
                 args.table, args.target, args.partition_col)   # Panel 05 logic

if __name__ == "__main__":
    main()
✓Driver included: the Microsoft SQL Server JDBC driver ships inside the Databricks Runtime — no Maven library install needed for the read itself. Only driver-side pyodbc needs an init script (unixODBC + msodbcsql18).
07

Deploying it: Databricks Asset Bundles

Asset Bundles (DABs) are the supported infrastructure-as-code wrapper: one databricks.yml declares jobs, clusters, and artifacts; the CLI validates, uploads, and runs them. Your entire ingestion framework becomes a git repo with a two-command deploy.

# databricks.yml
bundle:
  name: mssql-ingestion

resources:
  jobs:
    ingest_sales:
      name: ingest-sqlserver-sales
      tasks:
        - task_key: dim_customer
          spark_python_task:
            python_file: src/ingest_sqlserver.py
            parameters: ["--table", "dbo.DimCustomer",
                         "--target", "bronze.sales.dim_customer",
                         "--partition-col", "CustomerKey"]
          job_cluster_key: ingest_cluster
      job_clusters:
        - job_cluster_key: ingest_cluster
          new_cluster:
            spark_version: 15.4.x-scala2.12
            node_type_id: Standard_D4ds_v5
            num_workers: 4

targets:
  dev:
    workspace:
      host: https://adb-xxxx.azuredatabricks.net
The loop

git commit
→ databricks bundle validate
→ databricks bundle deploy -t dev
→ databricks bundle run ingest_sales -t dev

Targets (dev/prod) swap workspaces, cluster sizes, and catalogs without touching code.

Wheel variant

For anything beyond one script, declare a python_wheel_task with package_name + entry_point and list the wheel under libraries. DABs builds and uploads it on deploy. Your table manifest (YAML/JSON) ships inside the package; the entry point fans out with the Panel 03 ThreadPoolExecutor.

Why wheel > script for migrations

Unit-testable modules, versioned releases, shared helpers (bounds discovery, watermarking, DQ checks), and one job definition parameterized per table instead of 40 copies of a script.

Local development: Databricks Connect

Develop and debug the same .py files from VS Code or a JetBrains IDE — breakpoints, tests, refactoring — while execution happens on the cluster over Spark Connect.

pip install databricks-connect

# same ingestion code, local IDE, remote execution
from databricks.connect import DatabricksSession
spark = DatabricksSession.builder.getOrCreate()

# portable secrets (works locally AND in jobs):
from databricks.sdk import WorkspaceClient
pwd = WorkspaceClient().dbutils.secrets.get(
        "sqlserver", "password")
Caveat

DBUtils(spark) from pyspark.dbutils does not work over Spark Connect. Standardize on WorkspaceClient().dbutils and the same module runs unchanged in local dev, classic job clusters, and serverless.

Practical split

Iterate on one small table locally via Connect → push → bundle deploy → run the full manifest as a wheel job on a job cluster. No notebook ever enters the workflow.

09

The managed alternative: Lakeflow Connect for SQL Server

If the goal is ongoing replication rather than a one-time historical copy, Databricks now ships a fully managed SQL Server connector (GA since Sept 2025) — no custom pipeline code at all. It reads the source's change logs instead of hammering tables with queries.

How it works

An ingestion gateway (classic compute near the source) captures snapshots + changes into UC-governed staging storage; a serverless ingestion pipeline applies them to Delta tables. Uses Change Tracking for tables with primary keys (lighter on the source) and CDC otherwise; SCD Type 2 history is supported.

Coverage & requirements

Supports Azure SQL Database, Azure SQL Managed Instance, SQL Server on VMs, and on-prem via ExpressRoute / Direct Connect. Requires Unity Catalog + serverless enabled, CT/CDC enabled on the source (SQL Server 2012+), and a dedicated low-privilege ingestion login.

Operational caveats

The gateway should run continuously — if source log retention expires between runs, changes are lost and affected tables need a full refresh. Timestamps land in UTC. One-way only (no reverse ETL).

Where custom code still wins

One-shot historical backfills, exotic typing/transform logic during the copy, sources where you can't enable CT/CDC, and cost-sensitive bulk loads where a tuned partitioned JDBC job on a small cluster is cheaper than a continuous gateway.
